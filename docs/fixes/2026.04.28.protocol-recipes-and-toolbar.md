# Protocol recipes + toolbar discoverability
Date: 2026-04-28
Scope: turn Prism's slash commands into named, orchestrated "protocols" that
non-native (vibe-coder) users can discover and run from a toolbar menu, so the
quality of analysis no longer depends on the user knowing the right magic word.
Companion to `docs/fixes/2026.04.28.reviewer-methodology.md` — that doc tunes
the engine (verifier rules, `/review`); this doc wraps the engine in a UX
surface and sequences it.

## Problem
1. Today, high-quality diagnostics in Prism are gated by the user typing the
   right slash command (`/audit`, `/build`, `/verify`, etc.). Native CLI users
   are fine. Vibe coders never type these — competitor products in this niche
   don't ask for terminal-style commands at all.
2. Even native users have to know which commands to run, in what order, to
   accomplish a goal like "review the last refactor for cohesion" or "batten
   down the hatches on this app before I ship it."
3. The `reviewer-methodology.md` work fixes the engine so that the right
   behavior is the default (no magic word required). Once that lands, the
   commands are good — but they're still invisible.

## Goal
Make the existing slash functions runnable as **named protocols** through a
toolbar menu. Each protocol is an ordered recipe of slash commands plus a
checklist injected into the agent context. One protocol = one click =
one consolidated right-pane report.

Non-goals for this doc: implementing new analysis capabilities. The recipes
chain things Prism can already do (after the methodology pass lands).

## Architecture

### Recipe data model
A recipe is a static definition — JSON or a TypeScript const, source-controlled,
not user-authored at first. Shape:
* `id`: stable string, e.g. `"refactor-cohesion-review"`.
* `label`: user-facing name, e.g. `"Refactor Cohesion Review"`.
* `blurb`: one-line description shown in the toolbar menu and tooltip.
* `category`: `"review" | "security" | "ship" | "wiring"` (drives grouping
  and the toolbar icon).
* `steps`: ordered array of step objects:
  * `kind`: `"slash"` (run an existing slash command) or `"checklist"` (inject
    a checklist into the next agent turn) or `"shell"` (run a pnpm script,
    e.g. `pnpm typecheck`).
  * `command` / `args` / `script`: payload depending on `kind`.
  * `requiresApproval`: boolean. Default false for analysis steps, true for
    anything that writes files or runs network calls.
  * `onFailure`: `"abort" | "continue"`.
* `report`: how to consolidate output (see "Output surface" below).

Initial home: `src/recipes/index.ts` exporting a `RECIPES: Recipe[]` const.

### Recipe runner
A new module — proposal: `src/recipe-runner.ts` — exposes
`runRecipe(id: string, ctx: WorkspaceCtx, opts?: { signal?: AbortSignal, onProgress?: (ev: ProgressEvent) => void }): Promise<RecipeReport>`.
Responsibilities:
1. Resolve recipe by id. Fail loud if missing.
2. Emit a `planning` progress event with the full step list before any
   step runs, so the card can mount in its planning state.
3. For each step in order:
   * Emit a `step:active` event with the step index.
   * If `requiresApproval`, surface an approval card in the agent panel
     (reuse the existing approval-card component) and await user choice.
   * Dispatch to the existing slash-command handler / shell runner / agent
     query. Capture structured output into a per-step result object.
   * Emit `step:ok` or `step:failed` when the step settles.
   * Honor `signal.aborted` between steps; in-flight slash commands are
     allowed to finish (best-effort cancellation).
   * On failure, honor `onFailure`.
4. Emit a `done` event and hand all step results to the report consolidator.

Constraints:
* The runner MUST go through the same code paths that slash commands use
  today — no parallel implementation. Step `kind: "slash"` should call the
  same dispatcher `workspace.ts` uses when the user types `/audit`.
* The runner MUST NOT spam separate notice lines into AgentView for
  progress. Progress flows through the `onProgress` callback into the
  report card (see "Card lifecycle" below). One card mutates in place.

### Output surface — report card inside AgentView
Decision: render the report as a self-contained card mounted inline in
AgentView's message list, modeled on the existing Approval card component.
No new pane, no new split. The recent panel-split rebuild already paid
that tax once; another split is risky for marginal gain.

The card has its own component (`ProtocolReportCard`), its own CSS class
(`.protocol-report-card`), and its own internal state. It scrolls with the
turn that produced it, so re-reading a past recipe run later in the chat
just works. Persisted to `~/Documents/Prism/Reports/<recipe-id>-<ts>.md`
so the user can reopen / share outside the app.

### Card lifecycle
One card, three lifecycle states. The runner drives the transitions via
`onProgress` events; the card re-renders in place — it never gets
replaced or duplicated.

1. **Planning** (mounted on `planning` event, at run start)
   * Header: recipe `label` + timestamp + git HEAD.
   * Body: full step list as bullets, each showing the step's short
     label (e.g. "Run /audit", "pnpm typecheck", "Helper-body checklist").
   * No results visible yet. Active step gets a spinner once the first
     `step:active` arrives.
   * Footer: a `Cancel` button wired to the runner's `AbortController`.
2. **Running** (per `step:active` / `step:ok` / `step:failed` event)
   * Active step's bullet → spinner.
   * Completed step's bullet → ✓ (ok) or ✗ (failed). Failed steps with
     `onFailure: continue` keep the run going; failed steps with
     `onFailure: abort` end the run and the card transitions to `done`
     with the remaining steps left as pending bullets.
   * Per-step output is buffered into the card's internal state but not
     yet rendered as accordion sections.
3. **Done** (on `done` event)
   * Step list collapses into a single compact summary line:
     `"5/5 steps · 3 findings · 12.4s"`. Click the summary line to
     expand the step list back open.
   * Severity summary takes over as the main body, with counts per
     severity (mirrors `/audit`'s top section).
   * Per-step results render as a collapsible accordion; sections with
     findings default open, clean sections default closed.
   * Footer: "Open full report" link → on-disk markdown copy. "Re-run"
     button → calls `runRecipe(id)` again.

### Runtime types (sketch)
```ts path=null start=null
type StepState = "pending" | "active" | "ok" | "failed";
type RunStatus = "planning" | "running" | "done" | "aborted";

type ProgressEvent =
  | { kind: "planning"; steps: StepDef[] }
  | { kind: "step:active"; index: number }
  | { kind: "step:ok"; index: number; result: StepResult }
  | { kind: "step:failed"; index: number; error: unknown }
  | { kind: "done"; status: RunStatus };

interface RecipeReport {
  recipeId: string;
  startedAt: string;
  finishedAt: string;
  gitHead: string;
  status: RunStatus;
  steps: Array<{ def: StepDef; state: StepState; result?: StepResult }>;
  findings: Finding[]; // flattened severity-tagged findings across steps
  reportPath: string;  // on-disk markdown copy
}
```

### Toolbar UI
Single new toolbar button — proposal: shield-with-checkmark glyph,
labeled "Protocols" on hover. Clicking opens a menu:
* Grouped headings by `category`.
* Each entry shows `label` + `blurb`.
* Selecting an entry calls `runRecipe(id)`.
* Disabled entries (e.g. recipes that need a saved chat to operate on)
  show a tooltip explaining why.

The menu should be implemented as a real CSS class (no inline styles) and
support `Escape`-to-dismiss and viewport-edge clamping — the same lessons
from the tab context menu loose-end (`#7` in `recent-commits-loose-ends.md`)
apply here.

## Initial protocol catalog
Build these four first. Each maps to a real user goal and exercises the
runner's step kinds.

### 1. Refactor Cohesion Review (`category: review`)
Goal: after a batch of commits, did the work stay cohesive — did anything
leak across the rule lines a recent refactor drew?
Steps:
1. `slash`: `/review` (the new command from `reviewer-methodology.md`
   Phase 2) over the last N commits. N defaults to "since branch base".
2. `shell`: `pnpm typecheck`.
3. `shell`: `pnpm test` (if a test script exists).
4. `checklist`: inject the three methodology rules from
   `reviewer-methodology.md` Phase 1 (invariant grep, helper-body
   inspection, schema round-trip).
5. Consolidate.

### 2. Harden (`category: security`)
Goal: anti-hacker / batten-down-the-hatches sweep. Pitched at vibe-coder
apps that ship with wiring gaps.
Steps:
1. `slash`: `/audit` with security profile (existing).
2. `checklist`: secrets-in-source check, missing-auth-guard check,
   open-CORS / open-CSP check, dependency-CVE check via `pnpm audit`.
3. `shell`: `pnpm audit` — captured into the report.
4. Consolidate, severities boosted (a "Medium" from `/audit` becomes "High"
   under Harden's lens).

### 3. Pre-Ship Check (`category: ship`)
Goal: a vibe coder is about to deploy. Catch the embarrassing stuff.
Steps:
1. `shell`: `pnpm typecheck`.
2. `shell`: `pnpm build`.
3. `shell`: `pnpm test`.
4. `slash`: `/audit` with a "ship-blockers only" filter (new flag on
   `/audit`).
5. `checklist`: env-var presence check, console.log leftovers, TODO/FIXME
   density.

### 4. Wiring Gap Audit (`category: wiring`)
Goal: the niche we found ourselves — recently-wired code where the symbol
exists but the body is a stub or no-op.
Steps:
1. `checklist`: every helper called by code touched in the last 20
   commits — open the body, confirm it isn't a stub. (This is the
   `expandTilde`-class bug.)
2. `slash`: `/audit` scoped to recently-touched files.
3. Consolidate, with "stub" findings called out as their own severity.

## Phasing
Strict order — each phase delivers value standalone.

### Phase A — engine (prereq, lives in reviewer-methodology.md)
* Phase 1 of `reviewer-methodology.md`: tune verifier prompt.
* Phase 2 of `reviewer-methodology.md`: ship `/review`.
This doc assumes A is done.

### Phase B — recipe runner (no UI yet)
* `src/recipes/index.ts` with the four recipes above defined as data.
* `src/recipe-runner.ts` with `runRecipe()`.
* Wire a hidden slash command `/protocol <id>` that calls `runRecipe`. This
  lets us test the orchestrator end-to-end before adding UI.
* Consolidated markdown report written to disk.
Acceptance: `/protocol harden` from the prompt bar runs all Harden steps,
writes a report, prints a top-line summary in AgentView.

### Phase C — consolidated report pane
* `ProtocolReportView` (or the AgentView card variant — pick during
  implementation).
* Sections, collapsible accordion, severity summary.
Acceptance: re-running `/protocol harden` renders structurally instead of
as a wall of text.

### Phase D — toolbar menu
* Toolbar button + menu component.
* Wired to `runRecipe`.
* Keyboard, viewport-clamping, CSS class (lessons from tab-context-menu
  loose-end).
Acceptance: a user who has never typed a slash command can right-click
nothing, just click the toolbar shield, pick "Harden", and get a report.

### Phase E — FAQ / "what does this do?" affordance
Lower priority. Either:
* Tooltip on each menu entry (the `blurb`), OR
* A small "?" link inside the report header that opens a help drawer
  describing the steps the recipe ran.
Skip the dedicated FAQ right-pane — vibe coders won't read it.

## Acceptance checks (whole feature)
* A user with no CLI familiarity can complete a Harden pass without typing
  a slash command.
* Each recipe produces an on-disk markdown report under
  `~/Documents/Prism/Reports/`.
* Adding a new recipe is one file edit (`src/recipes/index.ts`) plus
  optionally a new icon — no runner changes.
* The toolbar menu reuses CSS classes; no inline styles; supports `Esc`
  and viewport-edge clamping.
* Running a recipe never bypasses the existing slash dispatcher (no
  parallel implementations of `/audit` etc.).

## Risks and mitigations
* **Token cost.** Long checklists injected per step balloon the context.
  Mitigation: checklists are injected ONLY for the steps that need them,
  and the runner trims earlier step output before later steps run.
* **Approval fatigue.** If every step prompts, the vibe coder bails.
  Mitigation: default `requiresApproval: false`; only file-writing or
  network steps prompt.
* **Recipe drift.** Slash commands evolve; recipes go stale. Mitigation:
  recipes reference slash commands by id, not by re-implementing them.
  When a slash command is removed, a startup check warns about recipes
  that reference it.
* **Discoverability of the toolbar button itself.** A new icon in a busy
  toolbar may go unnoticed. Mitigation: first-run nudge — small popover
  pointing at the shield for new workspaces.

## Out of scope
* User-authored recipes. The data model leaves room, but v1 is
  source-controlled only.
* Cross-project recipe sharing.
* Auto-running recipes on save / on commit. Manual-trigger only.
* Replacing the slash commands. Slash commands stay as the power-user
  surface; recipes are an additional layer on top.

## Why this is its own spec
`recent-commits-loose-ends.md` is concrete fix work for shipped regressions.
`reviewer-methodology.md` tunes the analysis engine itself. THIS doc is
the productization layer that turns the tuned engine into a discoverable
feature non-CLI users will actually use. Each is a separate review/QA
loop and a separate session's worth of work.

Order of operations across the three docs:
1. Land `recent-commits-loose-ends.md` items #2, #3, #4, #5 (cleanup of
   shipped regressions, fast wins).
2. Land `reviewer-methodology.md` Phase 1 (verifier prompt rules).
3. Land `reviewer-methodology.md` Phase 2 (`/review`).
4. Land THIS doc Phase B (recipe runner, hidden slash).
5. Land THIS doc Phase C (consolidated report).
6. Land THIS doc Phase D (toolbar menu) — at this point the feature is
   shippable to vibe coders.
7. Optional: THIS doc Phase E, plus the rest of `recent-commits-loose-ends`.
