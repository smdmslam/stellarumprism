# Recipe shell-step applicability
Follow-up surfaced by a real `/protocol pre-ship-check` failure: when
the user's shell cwd was a project that doesn't have a `typecheck`
script in its `package.json` (in this case `Development/OM`, an npm
project that uses `npm run lint` instead), pnpm exited 254 with
`ERR_PNPM_NO_SCRIPT  Missing script: "typecheck"` and the recipe
aborted at step 1.

The 254 was correct \u2014 pnpm did the right thing. But the recipe model
was wrong: it assumed every project has the same script names
(`typecheck`, `build`, `test`, `audit`). That's true for Prism itself
and for many TS / Node projects, but not universal. A user running
the toolbar's "Pre-Ship Check" against any project should not see a
fast 254 from step 1; they should see something like:

> Pre-Ship Check skipped \u2014 this project doesn't have a `typecheck`
> script. Add one to `package.json` or pick a different recipe.

OR the runner should pre-flight, detect the gap, and fail at the
recipe level (not at step 1) with a clear, actionable message.

## Status
Not started. Phase B `/protocol pre-ship-check` works fine on
projects whose scripts match (e.g. StellarumPrism); fails noisily
on projects whose scripts don't.

## Proposed fix

### 1. Add a `requires` shape to `StepDef` (shell variant)
For shell steps, allow the recipe to declare which `package.json`
scripts must exist. The runner reads `package.json` once per recipe
run and validates ahead of time.

```ts path=null start=null
{
  kind: "shell",
  label: "pnpm typecheck",
  script: "typecheck",
  // Implicit: `requires: [\"typecheck\"]` (defaults to [script]).
  // Override only when the step depends on more than its primary script.
}
```

Default behavior: `requires` is just `[script]`, so the existing
recipes pick up applicability checks for free.

### 2. Pre-flight at recipe start
Before running step 0, the runner:
1. Reads `<cwd>/package.json` (new Tauri command or reuse existing
   `read_file_text`).
2. Collects every `requires` entry from the recipe's shell steps.
3. For each missing script, decides per-step:
   * `onMissing: \"abort\"` (default) \u2014 the whole recipe aborts
     before any step runs, with a clear top-line: "Recipe
     'pre-ship-check' requires scripts 'typecheck', 'build', 'test'
     in package.json; this project has 'lint', 'dev'. Add the missing
     scripts or pick a different recipe."
   * `onMissing: \"skip\"` \u2014 the missing step transitions directly
     to `state: \"skipped\"` with a clear note; remaining steps run.
4. Slash steps are unaffected (they don't need package.json).

### 3. Recipe-level applicability summary in the report
The on-disk Markdown report should call out skipped / unmet steps in
the header so a user reopening the report can see "Recipe partially
applied: 2 skipped, 2 ran" at a glance.

### 4. Toolbar UX (Phase D dependency)
When the toolbar menu is built, recipes with unmet requirements in
the current cwd should render as **disabled** entries with a tooltip
explaining why ("requires 'typecheck' script in package.json"). This
turns a runtime error into a discoverable affordance.

## Acceptance checks
* `/protocol pre-ship-check` in StellarumPrism still runs cleanly
  end-to-end (no regression).
* `/protocol pre-ship-check` in a project without `typecheck` /
  `build` / `test` scripts emits a single recipe-level error BEFORE
  any spawn, naming the missing scripts. No `exit 254` mystery.
* `/protocol harden` works in any project that has `pnpm audit`
  (most do); unaffected by the typecheck gap.
* The on-disk report header shows applicability state.
* Phase D toolbar menu (when it lands) disables inapplicable recipes
  with a tooltip.

## Risks / open questions
* **Non-pnpm projects.** OM uses npm, not pnpm. The runner currently
  hard-codes `pnpm` in `run_pnpm_script`. A future "package-manager-
  aware" runner could detect the lockfile (pnpm-lock.yaml,
  package-lock.json, yarn.lock, bun.lockb) and choose the right
  command. Out of scope for this follow-up; track separately if it
  becomes a real need.
* **Workspace monorepos.** A recipe might need to run scripts in a
  workspace package, not the root. The current `cwd = workspace.cwd`
  assumption breaks. Same out-of-scope flag.

## Out of scope
* Replacing pnpm with a package-manager-aware runner.
* Auto-creating missing scripts.
* Cross-recipe dependency graphs (e.g. "Pre-Ship Check requires
  Refactor Cohesion Review to have run first").
* Project-specific recipes (a future feature where a project's
  `.prism/recipes/` could override the built-in catalog).

## Estimated effort
~2 hours. Most of it is the pre-flight package.json read + the
applicability summary in the report. Toolbar disabling rides on
Phase D and adds maybe 30 minutes there.

## Why this is its own doc
The script-applicability gap is orthogonal to:
* Phase B (recipe runner mechanics) \u2014 already shipped.
* Phase C (lifecycle card) \u2014 same renderer regardless of which steps
  ran or were skipped.
* Phase D (toolbar) \u2014 will *use* applicability state for disabled
  entries but doesn't have to ship with it.

Picks up cleanly in a fresh session as a 1-PR-sized change.

## Related session notes worth tracking
* The conversational pass agent that diagnosed the OM failure read
  the lint output and concluded the failure was about ESLint /
  React-Hooks rules. That conclusion was wrong \u2014 ESLint never ran;
  pnpm aborted before it. Worth keeping as an example of the model
  trying too hard to interpret a `pnpm exit 254` against a separate
  signal it had access to. Not an immediate fix item, but a
  data point for future methodology-prompt tuning.
* The `/protocol` slash-command registration drift (commit `4d2fbd2`)
  \u2014 added a slash dispatch in `workspace.ts` without registering it
  in `slash-commands.ts`. Worth adding "register in
  `slash-commands.ts`" as an explicit acceptance check on any future
  slash-command add, or a methodology check that flags new
  dispatches without matching popup entries.
