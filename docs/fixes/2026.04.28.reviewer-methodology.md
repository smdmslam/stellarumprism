# Improve Prism's reviewer methodology
Date: 2026-04-28
Scope: how Prism reviews recent work, where its current pass falls short,
and what to add so reviews catch the harder-to-spot regressions.

## Methodology gaps
Prism's pass nails the easy half: builds, tests, greps for unconsumed
identifiers, runs typecheck (or tries to). It doesn't reach the harder
half:

### Gap A — Cross-commit invariant tracking
Commit `db5c9d8` declared "xterm = shell only" as a new architectural
rule. The 17 commits that followed didn't all honor it (76 violations
in `workspace.ts`). Reviewing commits in isolation can't catch this —
you have to:
1. Lift the invariant from the refactor commit's message + diff.
2. Identify the OLD pattern (`this.term.write` for non-shell content).
3. Grep the whole repo for remaining occurrences.
4. List every violation.

### Gap B — Helper body inspection
`expandTilde` exists, has the right signature, has callers. A grep-based
audit reports "wired up". Reading the body shows it's a stub.

Reviewers need to OPEN the body of every helper called by recent code,
not just verify the symbol resolves.

### Gap C — Schema round-trip checking
When the frontend persists a new field through Tauri to Rust, the Rust
struct may not have that field. Serde silently drops unknown fields by
default (or rejects, depending on config). The pass should:
1. Diff the frontend's persisted shape against the Rust struct.
2. Flag any field on one side that's missing on the other.

## Where to put the methodology fixes
Three candidate homes — pick based on whether the rule is universal,
mode-specific, or domain-specific:

### 1. Verifier system prompt (recommended for universal rules)
Prism already runs the verifier model (`verifier_enabled` setting) as a
second-pass review on every turn. Universal rules belong here:
* "After any architectural-refactor commit (look for keywords like
  'split', 'move', 'retire'), grep the codebase for the OLD pattern."
* "For each helper function called by recent code, open its body and
  confirm it isn't a stub or a no-op."
* "If a frontend type adds a field that gets persisted, verify the
  matching Rust struct has the field."

Always-on. No user toggle. Doesn't bloat the main turn's token budget.

### 2. Mode persona for `/review` (or extend `/audit`)
A dedicated review mode whose system prompt includes the methodology
checklist explicitly. Useful when the user runs an explicit review pass
(like the 18-commit review here) and wants the agent to do more than
the default Grounded-Chat pass.

* Today there's `/audit` (security/code-quality findings on a scope) and
  `/build` (feature implementation). A `/review` mode would specifically
  target "is the recent work cohesive — did anything leak across the
  rule lines a recent refactor drew?"

### 3. Skills (only after they're actually wired in)
Right now skill files in `.prism/skills/*.md` are inert. The Settings
UI toggles enabled/disabled flags in localStorage, but nothing reads
those flags or injects skill content into the agent's context.

Skills make sense for project-specific or domain-specific knowledge
(e.g., "this Stripe project's webhook conventions") that not every user
needs. They are NOT the right home for universal review rules — that
belongs in the verifier prompt.

If skills are to ever do anything, the wiring is:
* `agent.ts query()` reads `settings.getEnabledSkills()` (currently
  `enabledSkills: Record<string, boolean>` in `AppSettings`).
* For each enabled skill, read the file content.
* Concat into `systemPrefix` or pass as context.
* Truncate / dedupe so it stays under a token budget.
* Optional: auto-attach by topic match against the prompt or mode, so
  users who never toggle anything still get relevant skills.

## Proposed plan

### Phase 1 — Tune the verifier prompt (universal rules, always-on)
* Open the Rust file that builds the verifier system prompt (likely
  `src-tauri/src/agent/verifier.rs` or wherever the verifier persona
  lives).
* Append three checklist items:
  1. "If the work being reviewed includes refactor commits (commits
     that move responsibility from one surface to another, retire a
     module, or change an architectural boundary), re-grep the
     codebase for the OLD pattern across ALL files. Flag every
     remaining occurrence."
  2. "For each helper function called by recent code, OPEN ITS BODY.
     Confirm the implementation does what the name claims. Stubs that
     return their input unchanged or no-op are common."
  3. "When a frontend type adds a persisted field, locate the
     matching Rust struct and verify the field exists. Flag schema
     mismatches."

### Phase 2 — Add a `/review` slash command + mode
* New slash command `/review [scope]` where scope defaults to "last 18
  commits" or accepts a count / git ref / file list.
* Mode persona seeded with the same three checklist items as Phase 1
  PLUS the request to:
  * Run `pnpm test`, `pnpm typecheck`, `pnpm build`.
  * Spot-check the most recent refactor commit's invariants.
  * Output a structured findings report (similar to `/audit` but for
    cohesion / unfinished-work, not security).

### Phase 3 — Actually wire skills into the agent context
Only worth doing if the user wants skill-based domain expertise to
work. Today it doesn't.

* `agent.ts query()`: assemble skill content from enabled-flagged files
  in `.prism/skills/`.
* Inject as `systemPrefix` (small per-skill content, dedupe).
* Decide UX: pure opt-in via Settings toggles (current direction) or
  auto-attach by topic match.
* Only after wiring is real, write actual skill files for things the
  user does want opt-in (e.g., a Stripe webhook checklist).

## Acceptance checks
Phase 1:
* Re-run a review on the same 18 commits (or any future refactor batch).
* The review must surface the "X violations of new invariant Y" pattern
  WITHOUT the user having to ask for it.
* Stub helpers must be flagged by name.

Phase 2:
* `/review` runs end-to-end and produces a findings report.
* Report includes the cohesion/invariant section by default.

Phase 3:
* Toggling a skill in Settings changes the agent's behavior on the
  next turn (verifiable by including a test skill that says "always
  include the word 'banana' in your response").

## Out of scope for this methodology pass
* Better diffing UI (the agent currently summarizes commits as text).
* Automated invariant detection (recognizing what an architectural rule
  IS from a commit message, without a human seed).
* Cross-project memory (skills today are per-project; no shared store).
* Performance: long verifier passes already cost tokens. If the
  checklist makes them slow, gate the harder rules behind `/review`
  mode rather than running on every turn.

## Why two specs and not one
The first spec (`recent-commits-loose-ends.md`) is concrete fix work the
user can hand to a Prism session as a checklist. Each item has a fix +
acceptance check. Estimated total: 4-8 hours.

This spec is a process change. Editing the verifier prompt and adding
a `/review` mode are infrastructure work that benefits all future
reviews — separate scope, separate session, different review/QA loop.

The shorter and more concrete one (loose ends) gets implemented first.
This one waits until the user is ready to invest in the methodology.
