## Phase C \u2014 ProtocolReportCard inside AgentView
Replace the current "many notice lines per recipe" rendering with one
self-contained card that lives inline in AgentView's message stream,
modeled on the existing Approval card pattern.

### What changes vs. Phase B
Today the runner calls `notify()` for every progress milestone:
```text
[protocol] starting "Harden" \u2014 2 steps
[protocol] step 1/2 \u2014 Security audit (working tree)
[protocol] step 1 ok (1m 23s)
[protocol] step 2/2 \u2014 pnpm audit (dependency CVEs)
[protocol] step 2 ok (4.1 s)
[protocol] "Harden" finished \u2014 2/2 steps ok \u00b7 1m 27s \u00b7 report: ...
```
That's six separate `<div class="agent-notice">` elements per recipe
run. Phase C replaces them with a **single** `ProtocolReportCard`
element that mutates in place across three lifecycle states.

### Lifecycle states
1. **Planning** (mounted on run start). Header: recipe label +
   timestamp + git HEAD. Body: full step list as bullets, each showing
   the step's short label. Active step gets a spinner once the runner
   reports it. Footer: `Cancel` button wired to the runner's
   `AbortController`.
2. **Running** (per step transition). Active step's bullet \u2192 spinner.
   Completed step's bullet \u2192 \u2713 (ok) or \u2717 (failed). Per-step output
   is buffered into the card's internal state but not yet rendered as
   accordion sections.
3. **Done** (on run resolve). Step list collapses into a compact
   summary line: `5/5 steps \u00b7 3 findings \u00b7 12.4s`. Click the summary
   line to expand the step list back open. Severity / pass-fail
   summary takes over as the main body. Per-step results render as a
   collapsible accordion; sections with output default open, clean
   sections default closed. Footer: "Open full report" link \u2192 on-disk
   markdown copy. "Re-run" button \u2192 calls `runRecipe(id)` again.

### Runner contract changes
* `runRecipe(id, ctx, opts?)` gains `opts?: { signal?: AbortSignal,
  onProgress?: (ev: ProgressEvent) => void }`.
* `ProgressEvent` shape:
  ```ts
  type ProgressEvent =
    | { kind: "planning"; steps: StepDef[] }
    | { kind: "step:active"; index: number }
    | { kind: "step:ok"; index: number; result: StepResult }
    | { kind: "step:failed"; index: number; error: unknown }
    | { kind: "done"; status: RunStatus };
  ```
* Runner stops calling `ctx.notify()` for per-step progress; the card
  is the renderer.
* Cancel: runner honors `signal.aborted` between steps; in-flight
  slash commands are allowed to finish (best-effort cancellation).

### AgentView changes
* New component `ProtocolReportCard` (likely a separate module so the
  styling stays scoped).
* New CSS class `.protocol-report-card` matching the Approval card's
  visual language (left bar, padding, rounded edge).
* The card mounts via a new `AgentView.appendCard(component)` hook or
  by directly appending into the current turn's children. Pick during
  implementation; Approval card already does the latter.

### Acceptance checks for Phase C
* Running `/protocol harden` produces ONE card in AgentView, not
  six notice lines.
* Step bullets visibly transition pending \u2192 spinner \u2192 \u2713/\u2717 as the
  recipe progresses.
* Cancel mid-recipe stops at the next step boundary; the card
  transitions to `done` with state `aborted`.
* Re-run button on a completed card kicks off the same recipe again
  in a new card below.
* Existing `/protocol` (bare, listing) still works as plain notice
  output.

## Phase D \u2014 Toolbar menu
Surface the recipes for users who never type slash commands.

### What gets added
* One new toolbar button (proposed glyph: shield-with-checkmark,
  hover label "Protocols"). Lives next to the existing layout-toggle
  buttons in the toolbar.
* Click \u2192 opens a small menu listing the recipes from `RECIPES`,
  grouped by `category` ("review" / "security" / "ship" / "wiring").
* Each menu entry shows `label` + `blurb` + a hover/title attribute
  for accessibility.
* Selecting an entry calls the existing runner via the same path
  the hidden `/protocol <id>` slash uses.

### UX requirements (carried over from the original spec)
* Real CSS class `.protocol-menu`; no inline styles.
* `Escape` keyboard listener dismisses the menu.
* Viewport-edge clamping so the menu never renders off-screen.
* Disabled entries (e.g. recipes that need a saved chat to operate
  on) show a tooltip explaining why \u2014 not strictly needed for the v1
  catalog but the runner should be safe to call anyway.

### Acceptance checks for Phase D
* A user with no CLI familiarity can run Harden by clicking the
  toolbar button and picking the entry.
* The menu reuses CSS classes (no inline styles), supports `Esc`,
  clamps to the viewport edges.
* Adding a new recipe to `src/recipes/index.ts` automatically
  surfaces it in the menu \u2014 no toolbar code change.
* The menu opens / closes via the same surface the tab context menu
  uses (already polished in commit `33b8483`).

## Suggested commit shape
1. Phase C \u2014 part 1: `ProtocolReportCard` component + CSS, mounted
   from a stub callback. No runner contract changes yet.
2. Phase C \u2014 part 2: extend `runRecipe` with `onProgress` +
   `AbortSignal`. Wire the card to the events. Drop the per-step
   `notify()` calls from the runner.
3. Phase D \u2014 part 1: toolbar button + menu shell (no items yet),
   keyboard + viewport polish.
4. Phase D \u2014 part 2: render `RECIPES` as menu entries grouped by
   category. Wire selection \u2192 `runRecipe`.
5. Optional Phase E: `?` link inside the report card opening a small
   "what does this recipe run" help drawer.

## Risks carried forward
* **Token cost during long recipes.** The `Done` state's accordion
  defaults to "open sections with findings, closed otherwise" so a
  recipe with five clean steps doesn't dump megabytes of stdout into
  the DOM at once.
* **Recipe drift.** Phase B already exposes the catalog as data; the
  toolbar should pick up new recipes automatically. Add a startup
  check that warns when a recipe references a removed slash command.
* **First-run discoverability of the toolbar button itself.** Mention
  in original spec; consider a small first-run popover pointing at
  the shield for new workspaces.

## Out of scope (still)
* User-authored recipes \u2014 v1 stays source-controlled.
* Cross-project recipe sharing.
* Auto-run on save / on commit.
* Replacing `/protocol` slash with the toolbar \u2014 the slash stays as
  the power-user surface; the toolbar is an additional layer.
