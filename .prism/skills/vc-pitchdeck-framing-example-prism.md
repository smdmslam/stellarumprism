# VC Pitch Deck Framing Example — Prism Case Study

Use this example alongside `./.prism/skills/vc-pitchdeck-framing.md` when the goal is to remember **how** a successful pitch-deck reframing happened in practice.

This file is not the general rulebook. It is the concrete case study showing how a technical, product-descriptive deck for Prism was turned into a much more investor-oriented deck.

---

## The starting situation

The original issue was not lack of substance. It was that the deck described the application more than it created investor desire.

The founder's diagnosis was effectively:
- the deck explains what Prism is
- it does not quickly make the investor feel why owning Prism matters
- it does not foreground value, strategic edge, or market position strongly enough
- it sounds more like product description than investment argument

This was the right diagnosis.

---

## The key founder prompt moves

A large share of the improvement came from the founder's framing, not from spontaneous model brilliance.

### 1. The weapon analogy
The founder introduced a high-value analogy:

> a weapon is not compelling because of the spec sheet; it is compelling because of what it can do

This mattered because it forced a shift away from feature inventory and toward consequence, effect, leverage, and buyer value.

Reusable lesson:
When a deck is too technical, ask:
- what does the buyer gain?
- what strategic advantage does ownership create?
- what becomes possible because this exists?

### 2. The Jobs / Apple constraint
The founder asked for a Steve Jobs style deck where each slide feels like one hard statement.

This constraint dramatically improved output quality because it:
- reduced verbal padding
- increased contrast
- pushed toward memorable phrasing
- discouraged feature dumping

Reusable lesson:
If the deck sounds generic, explicitly demand slide-level sharpness and punch.

### 3. The investor-language reframing
The founder emphasized:
- owning Prism should obviously create value
- Prism should feel like a competitive edge
- competitors should feel slow to catch up
- market size matters

This gave the model the correct diction: not product admiration, but strategic ownership logic.

Reusable lesson:
Ask directly for:
- value creation
- defensibility
- market size
- time-to-catch-up for competitors
- why this matters to an investor specifically

### 4. Pointing at the source materials
The founder told the model to examine:
- prior deck(s)
- auditor doc
- docs folder generally
- the internet

This mattered because it grounded the rewrite in real materials instead of generic deck priors.

Reusable lesson:
A good pitch-deck rewrite should synthesize from source material, not improvise from thin air.

## Approximate attribution of what produced the result

This breakdown is intentionally approximate. It is useful as a mental model, not as a scientific measurement.

### Part 1 — Prompting / founder framing (~50%)
The strongest directional moves came from the founder's instructions.

Examples:
- the weapon analogy
- the insistence on investor value over product description
- the Jobs-style slide constraint
- the explicit focus on competitive edge and catch-up time

Lesson:
A frontier model writes much better when the founder provides the worldview and the evaluative standard.

### Part 2 — Frontier-model pattern matching (~30%)
Once given the right genre and style constraint, the model contributed:
- aphoristic phrasing
- contrast-based headlines
- investor deck rhythm
- the classic wedge → platform → moat → ask flow

Examples from the successful framing:
- “Software creation accelerated. Verification did not.”
- “The editor should not grade its own homework.”
- “AI generates the code. Prism makes it safe to ship.”

Lesson:
Use the model for compression, cadence, and rhetorical architecture after the strategic framing is set.

### Part 3 — Prism's harness and context model (~20%)
Prism materially helped the quality of the result in two ways.

#### A. Whole-document context
The rewrite was informed by full source documents rather than embedding-retrieved fragments.
This allowed the deck to preserve the real wedge and substance of the company:
- incomplete AI refactors
- deterministic substrate
- multi-workflow architecture
- test-backed proof
- verification-layer positioning

Lesson:
Whole-document reads create more coherent strategic writing than fragment retrieval when the task requires an end-to-end company narrative.

#### B. Artifact discipline
The workflow naturally separated:
- analysis and discussion in chat
- final deck as a named artifact written to disk

Lesson:
A good harness improves not only model output quality but also process quality: discuss first, write second, preserve the result cleanly.

## What changed between the weak and strong deck

The winning transformation was not “add more facts.”
It was “reinterpret the same facts through an investor lens.”

### Before
The weaker deck led too early with:
- system specs
- substrate cell counts
- runtime and model-cost details
- IDE feature inventory
- implementation shape

This made the company feel technically interesting but did not maximize investor urgency.

### After
The stronger deck led with:
- the market shift
- the trust bottleneck
- silent failure in AI-generated code
- the separation of generation from verification
- Prism as a control point between code generation and production

This made the company feel more strategic, more necessary, and larger.

---

## The most important narrative shift

The decisive reframing was:

### Old framing
Prism is a technical AI development environment with a substrate, multiple tools, a workspace spine, and a shipped IDE surface.

### New framing
Prism is the verification layer for AI-built software.

This one change improved almost everything downstream.

Why it worked:
- it names a category
- it implies urgency
- it points to recurring spend
- it positions Prism near trust and production
- it sounds larger than a feature or plugin
- it naturally expands from wedge to platform

Reusable lesson:
Find the category-defining sentence that makes the product sound strategically inevitable.

## Specific deck moves that worked well

### 1. Start with the shift, not the product
A strong early line was:

> Software creation accelerated. Verification did not.

Why it worked:
- names a macro shift
- creates tension immediately
- makes the company feel like the answer to a structural imbalance

### 2. Name the failure vividly
A strong line was:

> AI code fails silently.

Why it worked:
- short
- emotionally charged
- easy to understand
- points directly at trust risk

### 3. Create a memorable thesis line
A strong line was:

> The editor should not grade its own homework.

Why it worked:
- aphoristic
- intuitive
- portable in conversation
- draws a clean distinction from existing AI editors

### 4. Define the company as a control point
A strong move was to say that Prism sits:

> between code generation and production

Why it worked:
- investors like control points
- it sounds infrastructural
- it links product value to workflow necessity
- it becomes more valuable as AI usage grows

### 5. Tie demand to the macro trend
A strong line was:

> Every AI-generated diff creates demand for verification.

Why it worked:
- explains why the market expands as AI expands
- turns AI adoption into a tailwind rather than a threat
- gives the company a scaling logic

### 6. Turn the wedge into a platform
A strong move was:
- wedge: incomplete AI refactors
- platform: trusted layer around AI software development

Why it worked:
- initial pain is sharp and real
- eventual company size is much larger than the wedge alone

## What should be reused next time

When writing a future VC deck from source materials, preserve this sequence:

1. Diagnose why the current deck feels descriptive rather than investable.
2. Ask what the investor should feel by slide 3.
3. Force the shift from “what it is” to “why owning it matters.”
4. Identify the macro shift that creates inevitability.
5. Identify the painful failure mode.
6. Draw the category boundary against adjacent tools.
7. Name the wedge.
8. Show the expansion path.
9. Convert technical architecture into moat language.
10. Keep slides sparse and declarative.

This sequence was a major reason the Prism rewrite succeeded.

## Reusable language patterns from the Prism rewrite

These patterns are worth reusing when they fit the facts.

### Category lines
- The verification layer for AI-built software.
- The second pass before production.
- The control plane for software trust in the AI era.

### Contrast lines
- Software creation accelerated. Verification did not.
- Others help create code. Prism helps trust it.
- Competitors built editors. We built the verification substrate.

### Outcome lines
- AI generates the code. Prism makes it safe to ship.
- Prism turns AI coding into a system teams can trust.
- Every AI-generated diff creates demand for verification.

### Expansion lines
- The wedge is verification. The platform is the trusted layer around AI software development.
- Start where the pain is sharpest. Then expand across the workflow.

## Cautions learned from the process

### 1. Strong claims should stay credible
Claims like these should be softened unless directly supported:
- nobody does this
- no competition exists
- impossible to copy
- every developer already uses AI

Use calm but strong alternatives:
- no clear category leader has emerged
- verification remains underserved
- most tools optimize for generation first

### 2. TAM should support, not dominate
In this case, the framing did most of the persuasive work.
The founder planned to revise TAM personally, which is reasonable.
Do not let weak or approximate market numbers drag down a strong narrative spine.

### 3. Technical founders often over-lead with implementation
This was the central risk in the original Prism deck.
The lesson is to delay implementation detail until after the investor understands the strategic value of the company.

## What this example proves about Prism itself

This pitch-deck session was not only a content exercise. It also demonstrated part of Prism's product claim.

Prism helped because:
- it could read the key source docs end-to-end
- it could preserve the output as a concrete artifact
- it encouraged a disciplined two-step workflow: discuss, then write

So the process of creating the better deck was itself evidence for one Prism thesis:

> comprehensive context + disciplined artifact creation produce better high-level work than fragmentary retrieval and one-shot drafting

That is useful not only for pitch decks, but also for strategy docs, launch memos, competitive analysis, and investor materials more broadly.
