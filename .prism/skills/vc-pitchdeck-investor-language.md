Use this skill when writing or rewriting a VC pitch deck and the main challenge is not lack of facts, but lack of sharp investor-facing language.

## Purpose

Technical founders often have enough substance for a strong deck but present it in language that sounds:
- descriptive rather than strategic
- product-focused rather than investor-focused
- precise but not memorable
- informative but not persuasive

This skill helps translate company substance into language that signals:
- category potential
- strategic position
- market inevitability
- defensibility
- recurring value
- venture-scale possibility

---

## The core language shift

Move from:
- what the product does
- how the system works
- what the interface contains
- what tools are included

Toward:
- why the company matters
- what economic role it plays
- what control point it owns
- what becomes more true as the market grows
- why replacing it would be hard

A deck does not become investable by becoming more detailed.
It becomes investable by becoming more strategically legible.

## Preferred investor concepts

When the facts support it, prefer language from these families.

### 1. Control point language
Use when the company sits in a must-pass workflow position.

Preferred phrases:
- control point
- checkpoint in the stack
- control plane
- gating layer
- trust layer
- system of record
- operating layer
- verification layer
- infrastructure layer

Use these to communicate:
- indispensability
- recurring relevance
- workflow leverage
- strategic position

### 2. Market shift language
Use when the company benefits from a real external trend.

Preferred phrases:
- market shift
- structural change
- the bottleneck has moved
- the stack now requires
- demand increases as
- the market created the need before a category leader emerged
- the next wave
- this becomes mandatory as adoption rises

Use these to communicate:
- timing
- inevitability
- macro tailwinds

### 3. Wedge language
Use when the company starts with a narrow but painful use case.

Preferred phrases:
- wedge
- beachhead
- sharp initial use case
- start where the pain is sharpest
- narrow entry point, broad expansion path
- first use case, larger platform

Use these to communicate:
- disciplined go-to-market
- focus
- ability to win before broadening

### 4. Moat language
Use when explaining why the company is durable.

Preferred phrases:
- defensibility
- architecture-level advantage
- difficult to retrofit
- compounds with usage
- switching costs
- institutional memory
- built in the right order
- shared substrate / shared foundation
- accumulates proprietary workflow knowledge

Use these to communicate:
- durability
- asymmetry versus incumbents
- increasing returns from adoption

### 5. Outcome language
Use when translating features into buyer value.

Preferred phrases:
- reduces hidden risk
- enables trust at scale
- lowers the cost of verification
- turns adoption into something organizations can govern
- makes velocity safer
- reduces review burden
- converts AI output into production-ready work
- makes teams comfortable shipping faster

Use these to communicate:
- economic and operational value
- budget relevance
- practical urgency

## High-value contrast patterns

Contrasts create deck rhythm and memorability.
Use them heavily when the facts support them.

### Pattern 1 — X accelerated, Y did not
Examples:
- Software creation accelerated. Verification did not.
- Code generation scaled. Trust did not.
- AI increased output. Review did not keep pace.

Why it works:
- names the imbalance
- creates immediate tension
- sets up inevitability for the company

### Pattern 2 — Others do X. We do Y.
Examples:
- Others help create code. Prism helps trust it.
- Editors generate. Prism verifies.
- AI writes the code. Prism decides when it is ready.

Why it works:
- creates category boundary fast
- gives the company a distinct role
- is easy to remember

### Pattern 3 — The old way is structurally wrong
Examples:
- The editor should not grade its own homework.
- Generation and verification are separate jobs.
- The same agent that made the change is the wrong one to certify it.

Why it works:
- sounds like insight, not promotion
- attacks the structure of the incumbent approach
- is sticky in conversation

### Pattern 4 — The more X grows, the more Y matters
Examples:
- The more code AI writes, the more verification the stack requires.
- The more machine-generated software enters production, the more trust infrastructure becomes necessary.

Why it works:
- ties company value to market growth
- turns macro adoption into a tailwind
- helps explain venture scale

## Language upgrades: weak vs strong

### Weak: descriptive product language
- tool
- assistant
- feature set
- helps developers
- improves productivity
- smarter workflow
- AI-powered platform
- integrated experience

### Stronger: investor-relevant language
- layer
- control point
- system of trust
- recurring workflow checkpoint
- production gate
- trust infrastructure
- strategic workflow position
- platform built from a sharp wedge

### Example upgrades
Instead of:
> Prism is an AI coding assistant with multiple tools.

Prefer:
> Prism is the verification layer between AI code generation and production.

Instead of:
> Prism has a diagnostic substrate with several cells.

Prefer:
> Prism built a deterministic verification engine that competitors cannot easily retrofit onto generation-first products.

Instead of:
> Prism stores artifacts in markdown files.

Prefer:
> Prism compounds institutional memory inside the repo, creating portable but sticky workflow intelligence.

Instead of:
> Prism supports multiple models.

Prefer:
> Prism is model-agnostic, which reduces vendor risk and improves cost elasticity as the model market evolves.

## Preferred ways to discuss market size

Market language should support the story, not overpower it.

### Good market phrases
- sits at the intersection of multiple large markets
- participates in a large and growing budget category
- benefits from adjacent spend already embedded in engineering workflows
- grows with the adoption of AI-assisted software development
- turns an emerging workflow problem into recurring software spend

### Use with caution
- total addressable market of [very large number]
- every developer is a customer
- trillion-dollar opportunity

These can sound inflated unless handled carefully.

### Better framing pattern
1. name the market shift
2. name the adjacent budget lines
3. show why this company captures spend as that shift grows

Example:
> As AI-generated code volume rises, verification becomes a recurring spend category across developer tools, code review, testing, and software trust infrastructure.

## Preferred ways to discuss competition

The goal is not to deny competition.
The goal is to define the company's lane clearly.

### Good competition language
- adjacent tools solve neighboring problems
- most current products optimize for generation first
- verification remains underserved
- no clear category leader owns this layer yet
- existing incumbents approach this from a different architectural starting point
- competitors are optimized for a different primary job

### Avoid if unsupported
- no competitors
- nobody else is doing this
- impossible to copy
- winner-take-all already

### Strong comparison pattern
1. name the adjacent category
2. state its primary job
3. explain why your company's job is different

Example:
> PR review bots analyze changes near merge. Prism verifies AI-generated work before merge and against the live project state.

## Preferred ways to discuss moat

A moat slide should answer: why does this get stronger, stickier, or harder to replace over time?

### Strong moat ingredients
- built in the right order
- architecture incumbents cannot easily retrofit
- artifacts that accumulate over time
- workflow memory that deepens with use
- cross-surface reuse of the same substrate
- vendor resilience from model-agnosticism
- switching cost from embedded process or institutional memory

### Good moat phrasing
- Trust compounds.
- The more Prism is used, the harder it is to replace.
- Competitors built editors. We built the verification substrate.
- The product gets stronger as artifacts, patterns, and project memory accumulate.
- Every new workflow rides the same underlying substrate.

### Avoid
- moat = features
- moat = speed of shipping features alone
- moat = “we work hard”

A moat should sound structural.

---

## Preferred ways to discuss traction and proof

If traction is early, frame proof as de-risking rather than claiming victory.

### Good proof language
- product exists
- infrastructure is live
- multiple workflows are already powered by the same core engine
- dogfooding has surfaced real bugs and validated the need
- shipped product reduces platform risk
- the architecture is already carrying more than one use case

### Avoid
- overclaiming product-market fit
- overstating revenue or adoption if not strong
- trying to compensate for weak traction with inflated adjectives

### Good proof pattern
1. shipped
2. working
3. validated in real use
4. extensible beyond initial wedge

---

## Preferred ways to discuss the ask

The ask should sound proportional to the opportunity and the current stage.

### Good ask language
- raising to own the category
- accelerate product development
- expand into enterprise environments
- convert wedge adoption into broader workflow ownership
- win design partners and early strategic customers
- extend the existing substrate into higher-value workflows

### Avoid
- vague “scale the business” phrasing with no anchor
- overly detailed operating budgets inside the slide unless requested
- fundraising language disconnected from the roadmap

### Good ask pattern
1. what is being raised
2. what it funds
3. what that unlocks
4. why now is the moment to do it

These phrases often improve pitch-deck language when grounded in truth.

### Category-defining
- verification layer
- trust layer
- control plane
- production gate
- operating layer
- system of record for software trust

### Economic / strategic
- budget line
- recurring spend
- strategic edge
- high-value checkpoint
- must-pass layer
- compounding advantage
- embedded workflow position

### Expansion / ambition
- wedge into a broader platform
- narrow entry point, large eventual surface area
- adjacent workflows on the same substrate
- beachhead into a larger category

### Credibility / discipline
- evidence-backed
- deterministic
- grounded in real project state
- auditable
- calm but strong
- sharper, not louder

## Anti-patterns to remove from drafts

Delete or rewrite language that sounds like this:
- revolutionary AI platform
- empowers developers everywhere
- seamless integrated solution
- next-generation workflow assistant
- cutting-edge intelligence
- transformative synergy
- robust feature set
- best-in-class experience

These phrases usually weaken the deck.
They sound generic, vendorish, or unserious.

---

## A practical editing pass

When reviewing a draft, do a language-only pass and ask:

### Slide by slide
- Is this sentence memorable?
- Is it investor-relevant?
- Does it define a category, wedge, moat, or control point?
- Does it sound stronger if converted into a contrast?
- Is the language calm but sharp?
- Does it sound like infrastructure or like a tool demo?

### Rewrite checklist
- [ ] replace generic AI wording
- [ ] replace weak verbs with stronger ones
- [ ] compress long explanations into cleaner contrasts
- [ ] convert technical statements into strategic consequences
- [ ] remove unsupported absolutes
- [ ] make at least a few lines quotable
