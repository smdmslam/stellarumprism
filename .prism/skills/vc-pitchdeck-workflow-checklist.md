# VC Pitch Deck Workflow Checklist Skill

Use this skill as the operational checklist when creating, rewriting, or sharpening a VC pitch deck from product docs, prior decks, source materials, and market research.

This file is meant to be practical and reusable. It complements:
- `./.prism/skills/vc-pitchdeck-framing.md`
- `./.prism/skills/vc-pitchdeck-framing-example-prism.md`

Keep the `vc-pitchdeck-` filename prefix for related skills so they group naturally in folder listings.

---

## Goal

Produce a deck that makes the company feel:
- strategically valuable
- easy to understand quickly
- sharp in its wedge
- large in its eventual opportunity
- credible in its claims
- defensible in its position

The deck should not merely explain the product.
It should make an investor understand why the company matters.

---

## Phase 1 — Gather source material

Before writing, collect the best available inputs.

### Required inputs
- current or previous pitch deck
- main product positioning doc
- architecture / strategy docs
- competition notes if available
- traction / proof points if available
- pricing assumptions if available

### Optional inputs
- market research
- founder notes / raw thoughts
- customer quotes
- bug examples or compelling incidents
- roadmap or milestone plans

### Checklist
- [ ] locate the current deck
- [ ] locate the strongest product-positioning document
- [ ] locate strategy / competition docs
- [ ] locate any traction / shipped-product evidence
- [ ] locate pricing / business model assumptions
- [ ] decide whether fresh market research is needed

### Rule
If strong source docs exist, read them before drafting. Do not improvise a deck from memory when the repo already contains better material.

---

## Phase 2 — Diagnose the current deck

Before rewriting, identify why the current deck underperforms.

### Common failure modes
- too descriptive
- too technical too early
- weak investor language
- no clear category line
- no clear wedge
- no macro shift / why-now logic
- no moat framing
- too many features, not enough consequences
- TAM presented without narrative spine

### Questions to answer
- What does the deck currently explain well?
- What does it fail to make the investor feel?
- Does it describe the product more than the value of owning the company?
- Does it make the market pull obvious?
- Does it sound like software or strategy?

### Checklist
- [ ] identify what is strong in the current deck
- [ ] identify what is merely descriptive
- [ ] identify what should be cut, delayed, or reframed
- [ ] identify the missing investor-level message

---

## Phase 3 — Find the narrative spine

Do not draft slides until the narrative spine is clear.

### Core questions
Answer these in plain English first:

1. What changed in the world that makes this company matter now?
2. What painful failure or bottleneck does the company address?
3. Why are current alternatives structurally insufficient?
4. What exactly is the company's job in the stack or workflow?
5. What is the wedge?
6. How does the company expand beyond the wedge?
7. Why will this be hard to replace or slow to copy?
8. Why does demand increase as the market grows?

### Checklist
- [ ] define the macro shift
- [ ] define the painful failure mode
- [ ] define the category boundary
- [ ] define the wedge
- [ ] define the platform expansion path
- [ ] define the moat in business language
- [ ] define the clearest one-line company framing

### Rule
If you cannot explain the company in 2–3 strong sentences before making slides, the deck is not ready to draft.

---

## Phase 4 — Convert facts into investor language

Translate the source material into investor-relevant meaning.

### Translation table
- features → value
- tools → workflows solved
- architecture → defensibility
- implementation depth → barrier to replication
- pricing → recurring spend logic
- technical workflow position → control point
- product memory / artifacts → switching costs
- broad usage → market pull

### Checklist
- [ ] turn the product description into a value proposition
- [ ] turn the architecture into moat language
- [ ] turn the workflow into economic benefit
- [ ] turn the wedge into a beachhead for expansion
- [ ] turn market trends into a why-now argument

### Rule
Never leave an important technical fact untranslated. Ask: why does this matter to an investor?

---

## Phase 5 — Choose the slide structure

Once the spine is clear, define the slide order.

### Default strong sequence
1. cover / category line
2. market shift
3. painful problem
4. core insight
5. solution
6. value to users / buyers
7. economic or strategic significance
8. wedge
9. moat / why this wins
10. product surface / expansion
11. why now
12. market
13. competition
14. proof / traction
15. business model
16. vision
17. ask

### Checklist
- [ ] confirm the opening is category-first, not feature-first
- [ ] confirm the early slides create urgency
- [ ] confirm the wedge is visible
- [ ] confirm expansion is visible
- [ ] confirm moat appears before the ask
- [ ] confirm proof exists before big claims accumulate

### Rule
Not every deck needs the same number of slides, but the logical progression should remain intact.

---

## Phase 6 — Write the slides

When writing slides, optimize for clarity, force, and recall.

### Writing rules
- one main claim per slide
- minimal filler
- strong headlines
- concrete language
- avoid generic startup jargon
- use short punchy contrasts where possible
- keep details for later slides or notes

### If the desired style is Jobs / Apple
Use:
- short headlines
- clean contrasts
- one idea at a time
- rhythm and memorability
- no overcrowding

### Checklist
- [ ] each slide has one dominant point
- [ ] each headline could be spoken aloud cleanly
- [ ] each slide helps belief, not just comprehension
- [ ] technical details are delayed until they support proof or moat
- [ ] the deck reads quickly

### Rule
A good slide should feel quotable.

---

## Phase 7 — Review for investor relevance

After drafting, review slide by slide.

### Questions for every slide
- Does this increase investor conviction?
- Does this make the company sound more valuable, more necessary, or more defensible?
- Is this slide about value or just description?
- Would this still matter if the investor forgot the product details?

### Checklist
- [ ] cut any slide that is mostly brochure copy
- [ ] cut any redundant feature listing
- [ ] replace weak verbs with stronger ones
- [ ] simplify crowded slides
- [ ] ensure the company feels larger than the initial feature set

### Rule
If a slide is merely informative but not persuasive, rewrite or remove it.

---

## Phase 8 — Review for credibility

A strong deck is persuasive because it is sharp **and** believable.

### Claims that often need softening
- nobody does this
- no competition exists
- impossible to copy
- every developer uses AI
- all enterprises need this now

### Better alternatives
- no clear category leader has emerged
- verification remains underserved
- most tools optimize for generation first
- current alternatives are structurally aimed elsewhere
- the market need is emerging faster than category ownership

### Checklist
- [ ] remove brittle absolutes unless proven
- [ ] make competitive claims credible
- [ ] keep TAM broad but defensible if numbers are provisional
- [ ] separate proven facts from ambition

### Rule
Calm language often sounds stronger than hype.

---

## Phase 9 — Decide what belongs in notes vs. slides

Slides should stay sparse.
Use notes for:
- nuanced competitive explanation
- detailed market sizing logic
- technical proof details
- implementation specifics
- anticipated objections and answers

### Checklist
- [ ] move excess detail out of slides if it weakens punch
- [ ] preserve supporting arguments in notes or companion docs
- [ ] keep only the strongest visible points on slides

### Rule
Do not overcrowd the slide deck just because the source material is rich.

---

## Phase 10 — Final readiness check

Before declaring the deck done, verify the following.

### Final checklist
- [ ] the category line is strong
- [ ] the opening explains the macro shift
- [ ] the problem is painful and easy to grasp
- [ ] the insight is memorable
- [ ] the solution sounds distinct from adjacent categories
- [ ] the wedge is sharp
- [ ] the expansion path is visible
- [ ] the moat is clear
- [ ] the why-now timing is plausible
- [ ] the market is large enough to matter
- [ ] the proof points reduce risk
- [ ] the ask feels proportional to the plan
- [ ] the ending line is memorable

### Rule
If the founder still says “this explains the product, but doesn’t incite investment,” keep working.

---

## What to preserve after a successful session

When a deck rewrite works especially well, do not preserve only the deck.
Preserve the method too.

Save:
1. the finished deck
2. speaker notes if they exist
3. a general skill capturing the framing rules
4. a case-study skill capturing why this specific rewrite worked
5. any especially effective founder prompt language

### Checklist
- [ ] save the final deck as an artifact
- [ ] save the general method as a reusable skill
- [ ] save the successful case as an example skill
- [ ] preserve especially good prompt language or analogies

### Rule
The best pitch-deck sessions should compound into reusable institutional knowledge.

## Reusable founder prompts to keep nearby

These are useful starter moves when a deck needs upgrading.

### Prompt move 1 — value over description
> This describes the application, but it does not incite investment. Move the deck to why owning this company matters.

### Prompt move 2 — consequences over specs
> A product is compelling not because of its spec sheet but because of what it enables, what advantage it creates, and what it lets the buyer do.

### Prompt move 3 — sharp slide style
> Think Steve Jobs. Each slide should feel like one statement that hits immediately.

### Prompt move 4 — investor filters
> Make the wedge obvious, the market large enough, the moat credible, and the value creation clear.

### Prompt move 5 — discuss before writing
> First explain what is wrong with the current deck and what narrative spine it should have. Then draft the new one.

## Final lesson

A great VC deck is not assembled by adding more information.
It is built by arranging information in the order that creates belief.

This checklist exists to make that order repeatable.
