# Beta distribution & update plan (StellarumPrism)

**Goal:** Let someone clone the repo, get a running app with **visible shell / file tree / agent** without a long manual recipe, and **update often** as main moves.

**Context from feedback:** One tester hit a **long install command list**; after the app started they saw **no panes** (file explorer, agent, terminal). This plan treats both **onboarding** and **defensive UI** as first-class.

---

## 1. Current baseline (what works today)

| Step | Command / artifact |
|------|---------------------|
| JS deps | `pnpm install` |
| Desktop app (required for full UI) | `pnpm tauri dev` |
| API key | `~/.config/prism/config.toml` → `[openrouter] api_key = "..."` |

The README “Quick start” is short, but **prerequisites are implicit**: Node, **pnpm**, **Rust toolchain**, and **Tauri system deps** (platform-specific). Testers who skip or mis-order steps get confusing failures.

**Web-only `pnpm dev`** serves the Vite shell without the Tauri backend — not a supported beta path; layout and PTY features depend on the native app.

---

## 2. Likely cause of “app with no windows”

Without a reproduce on the tester’s machine, the highest-probability explanations are:

1. **Session restore with all panes hidden**  
   Tab layout persists per-tab visibility (`sidebar_visible`, `preview_visible`, `terminal_visible`, `agent_visible`, etc.) in `~/.config/prism/session.json` (via Rust `read_session_state`). If every flag was saved `false` (bug, migration, or extreme “hide all” usage), the workspace mounts with classes like `sidebar-hidden`, `terminal-hidden`, `agent-hidden` simultaneously → **empty-looking chrome**.

2. **Wrong launch mode**  
   Opening only the Vite dev URL in a browser instead of **Tauri** → incomplete or broken layout.

3. **Very small/zero effective pane sizes**  
   Less likely if clamping works, but corrupt `state.json` layout in the repo cwd could still feel “blank” until reset.

**Engineering follow-ups (recommended):**

- **Sanity invariant:** On workspace init or session restore, if *all* of sidebar / preview+terminal stack / agent are hidden, **force defaults** (or show a one-time toast: “Layout reset — use View menu / shortcuts to toggle panes”).
- **Beta doc:** One paragraph: “If the UI looks empty, reset session: quit app, rename `~/.config/prism/session.json` to `.bak`, relaunch” (or add a **Debug → Reset layout** action later).
- **Telemetry (optional):** Log layout flags at first paint in dev builds only.

## 3. Tier A — Fast wins (minimal code, high leverage)

| Item | Purpose |
|------|--------|
| **`scripts/bootstrap-dev.sh` (and `.ps1`)** | Single entry: check `node`, `pnpm`, `rustc`, `cargo`; run `pnpm install`; print “Run: `pnpm tauri dev`”. |
| **`docs/BETA-TESTER.md`** | One page: prerequisites links (Rustup, Tauri prereqs), clone → bootstrap → `pnpm tauri dev` → config.toml → screenshot of expected layout. |
| **Pin toolchain** | `package.json` `engines`, optional `rust-toolchain.toml` if the repo doesn’t already pin — reduces “works on my machine”. |
| **GitHub Actions: `ci.yml` already builds** | Add a **manual workflow** `build-desktop-artifacts` that runs `pnpm install` + `pnpm tauri build` on tag or `workflow_dispatch`, uploads `.dmg` / `.msi` / `.AppImage` as **draft** releases for testers who won’t compile. |

---

## 4. Tier B — “Easy install” for non-developers

| Track | Description |
|-------|-------------|
| **Prebuilt binaries** | Primary path for beta: download release asset, open app, add API key. No Rust required. |
| **Installer polish** | Code signing when ready (macOS notarization, Windows SmartScreen) — later phase. |
| **Version surface** | About box or status line showing **git sha / app version** so testers report bugs against a known build. |

---

## 5. Tier C — Easy updates (rapid iteration on `main`)

| Approach | Pros | Cons |
|----------|------|------|
| **TAURI updater + nightly / weekly channel** | True in-app updates | Needs signing + endpoint hosting |
| **Documented git loop** | `git pull && pnpm install && pnpm tauri dev` | Still dev-centric |
| **Release tags + GitHub Releases** | Testers pull `.dmg` when notified | Manual download each time |
| **Homemade script `scripts/update-dev.sh`** | `git pull`, `pnpm install`, optional `cargo build` in `src-tauri` | Doesn’t replace binary updates |

**Recommendation for the next few weeks:** ship **Tier A + manual Release artifacts** on each “beta checkpoint” tag; parallel-design **updater** once binaries are stable.

---

## 6. Acceptance criteria (beta-ready checklist)

- [ ] One **bootstrap script** + **BETA-TESTER.md** merged and linked from README.
- [ ] **CI or documented path** produces installable artifacts for at least **one** OS the team uses daily (e.g. macOS `.app`/`.dmg`).
- [ ] **Layout/session:** invariant or documented recovery if all panes hidden; verify fresh clone + first launch shows **sidebar + center + agent** by default.
- [ ] **Version identifier** visible for bug reports.
- [ ] **Update path** documented (git-based now; updater later).

---

## 7. Suggested order of execution

1. Reproduce or confirm **blank UI** with session.json containing all-false visibility → implement **layout sanity reset** (small PR).
2. Add **`scripts/bootstrap-dev.sh`** + **`docs/BETA-TESTER.md`** + README link.
3. Add **GitHub Actions** workflow to attach **Tauri build artifacts** to releases (or nightly).
4. Plan **Tauri updater** + hosting after stable signing story.
