# Notify over-correction — convert structural renderers to Markdown
Date: 2026-04-28
Scope: follow-up to commit `08d1976` (loose-end #1, term.write migration).
Caught by a `/review`-style conversational pass on the same 5 commits that
landed the migration — see `.prism/Review last 5 commits.md`.

## Background
Loose-end #1 routed ~76 `this.term.write` / `this.term.writeln` calls in
`src/workspace.ts` through two new helpers:
* `notify(message)` → `agentView.appendNotice("router", message)`
* `notifyError(message)` → `agentView.appendError(message)`

That cleared the architectural rule (xterm = shell only) in one pass.
But the loose-ends doc itself flagged two options for that migration:
* (a) strip ANSI inside `appendNotice` and let the panel render plain text
* (b) convert structural renderers (`/help`, `/last`, `/model`, audit /
  build reports) to emit Markdown/DOM and render via `appendProse` so
  headings, code, color, and tables land properly

The recommendation in the original doc was: **(b) for the structural
commands, (a) only for short notices like `[verify] off`.** The migration
took (a) for everything as step one, which is defensible (kills the
bleeding fast) but leaves a UX regression on the structured outputs.

## Findings from the conversational pass
1. **[warning] `src/workspace.ts:2106` — `notify()` funnels structured
   slash-command/report output through `appendNotice("router", …)`,
   which strips ANSI and sets textContent. Multi-line reports
   (`/models`, `/history`, `/help`, `/last`, audit summaries, build
   summaries) lose their structural / color cues. Evidence: notify
   call sites at lines 805, 819, 844, 856, 1635, 1719, 3480 in
   `src/workspace.ts`; `appendNotice` implementation at
   `src/agent-view.ts:242`.
2. **[info] `src/workspace.ts:3437` — `renderLoadedTranscript()`
   collapsed the entire loaded transcript into one giant notice block.
   It's coherent, but it bypasses the turn / prose / tool-card
   rendering AgentView already has via `beginTurn` / `appendProse` /
   `appendToolCall`, so loaded chats look visually different from
   live chats.

The first finding is a real UX regression. The second is mostly a
fidelity issue for the transcript-replay path.

## Proposed fix

### 1. Extend AgentView with a structured-report renderer
Either:
* New method `appendReport(markdown: string)` that goes through `marked`
  and renders into the message stream as a styled section — closely
  mirrors the existing `appendProse` plumbing, just emits at the
  agent-turn level rather than the per-token level.
* Or generalize: add `appendMarkdown(markdown: string, kind: "report" | "prose" | "review")` and let CSS scope styling per kind.

Recommend the first (`appendReport`) — small, additive, easy to
deprecate later if we generalize.

### 2. Convert the ANSI renderers piecemeal
Each is small and independently mergeable. Suggested order, easiest →
highest-value:
* `renderHelpAnsi` → `renderHelpMarkdown` (just a list of slash commands).
* `renderModelListAnsi` → `renderModelListMarkdown` (curated list with
  short aliases).
* `renderHistoryAnsi` → reuse existing prose rendering (each message
  becomes its own turn-shaped section).
* `renderLastAnsi` → small Markdown summary (audit + build pointers).
* `renderAnsiFindings` → Markdown variant (this is the highest-value
  conversion: a structured table of severity / file:line / description
  / fix prints far better as Markdown than as ANSI strip).
* `renderAnsiBuildReport` → same shape as findings.

Each call site in `workspace.ts` flips from `this.notify(renderXxxAnsi())`
to `this.agentView.appendReport(renderXxxMarkdown())`.

The legacy ANSI renderers can stay around (or be retired) as a separate
janitorial commit once the call sites are clean.

### 3. Preserve `notify()` for short status lines only
After the migration, `notify()` should ONLY carry single-line state
notices: `[verify] off`, `[files] sidebar → Files tab`, `[fix] applying
N of M`, `[edit] saved …`, etc. Anything multi-line or structured goes
through `appendReport`.

### 4. Fix transcript-replay rendering (the [info] finding)
Replace the current single-notice approach in `renderLoadedTranscript()`
with calls to `beginTurn` / `appendProse` / `appendToolCall` so a loaded
chat replays through the same surfaces a live chat uses. The result:
loaded chats are visually indistinguishable from live ones.

## Acceptance checks
* `/help`, `/models`, `/history`, `/last`, and `/audit` reports render
  as styled Markdown sections in the agent panel — headings, lists,
  inline code, tables all visible.
* `notify()` remains the only path for short status lines.
* Loading a saved chat replays each message into its own turn,
  including assistant tool-call cards from a v2 (`.full.md`) export.
* `pnpm typecheck` clean; `cargo check` clean.

## Out of scope
* Building a new ANSI-to-Markdown converter library. The existing
  ANSI renderers can be rewritten directly to emit Markdown — they're
  already string-builders.
* Replacing `appendNotice` itself. Short-status notices are still its
  job.
* Persisting the structured-report Markdown to disk. Audit / build
  reports already write Markdown sidecars under `.prism/`; this work
  only changes the in-app rendering surface.

## Why this is a separate commit / session
The previous commit (`08d1976`) closed the architectural rule
violation in one pass. This work is a UX upgrade on top of a
correctly-routed surface — orthogonal scope, separate review/QA loop,
and probably 2 hours of small per-renderer commits rather than one
big change. Cleaner history, easier reverts.

## Estimated effort
~2 hours total. Each renderer conversion is a small, independent
commit. Suggested commit shape:
1. Add `AgentView.appendReport(markdown)` + CSS for the new section kind.
2. Convert `renderHelpAnsi` → `renderHelpMarkdown`; flip the `/help`
   call site.
3. Same for `/models` and `/last`.
4. Same for `/history` (reuses existing prose rendering, so possibly
   even smaller).
5. Audit + build report Markdown variants — the meatiest, structured
   tables.
6. `renderLoadedTranscript()` rewrite through `beginTurn` /
   `appendProse` / `appendToolCall`.
