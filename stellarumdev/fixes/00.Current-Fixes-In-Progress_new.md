### 6) File next / previous — file viewer + reader

**Scope:** Add explicit navigation to the **next** and **previous** *file* (not directory) relative to the currently open path, in both the **center file viewer / editor** (`Workspace.openFileInEditor` / `.file-preview` header) and the **Reader** modal (`ReaderUI`, per active pane when split). Goal: fast linear review of sibling files without returning to the tree for every click.

**Ordering rule (MVP):**

- Resolve the **parent directory** of the current file’s absolute path.
- List **immediate children** of that directory that are **files** (exclude subdirectories from the navigation sequence).
- Sort by **lexicographic / locale-aware filename** (match OS or match existing tree sort — pick one and document it; consistency with the sidebar file tree is ideal).
- **Current file** = index `i`; **Next** = file at `i + 1`, **Previous** = `i - 1`.
- If the current path is not under `cwd` or the file no longer exists, treat as **no navigation** (disable controls or no-op with optional toast).

**Hidden / ignored files:**

- Respect the same **show hidden** policy as the file tree (if the tree hides dotfiles, navigation skips them; if shown, include them). Avoid inventing a third rule.

**Dirty buffer:**

- Reuse the **existing** “open another file while dirty” behavior (confirm discard / save flow) already used when switching files in the viewer. Reader panes that are editable should follow the same rules as today for `loadPane` / save.

**UI / affordances:**

- **File viewer:** Two small icon buttons in `.file-preview-header` (e.g. chevron up/down or single “stack” icon with dropdown — prefer **two buttons**: previous / next) with `aria-label`s (`Previous file in folder`, `Next file in folder`) and `title` tooltips including shortcut if any.
- **Reader:** Same control pair on the **active** pane’s toolbar row (mirror copy-path / copy-contents placement in `reader-ui.ts`). In **split** mode, navigation affects **only the focused pane** (left vs right), not both.
- **Disabled state:** Previous disabled at first file in list; next disabled at last file. Optional **wrap** is out of scope for MVP unless trivial; if wrap is added later, make it a Setting.

**Keyboard (optional follow-up, same sprint if cheap):**

- e.g. `Alt+↓` / `Alt+↑` or chord that does not collide with existing `tabs.ts` / editor bindings — only add if a quick audit shows a free pair; otherwise ship icons-only first.

**Implementation sketch (for whoever implements):**

- Prefer one shared helper (TS): input `absolutePath` (or `cwd` + relative), `direction: 1 | -1`, output `nextAbsolutePath: string | null` (or `{ path, cwd }` as needed). Implementation lists parent once via existing Tauri surface (`list_dir_entries` / `list_directory_tree` one-level — reuse what already returns sorted, filter `kind === "file"`, apply hidden filter).
- **Workspace:** After resolving path, call existing `openFileInEditor(nextPath)` (or equivalent) so diagnostics, meta line, dirty handling stay centralized.
- **Reader:** `loadPane(activeSide, cwd, nextPath)` after resolve.
- **Tests / manual QA:** Single-file dir (both disabled except no-op); two files (toggle ends); dirty file + next (confirm dialog); hidden file toggling; reader split with different folders per pane (each pane’s “next” is **its** file’s parent only).

**Acceptance:**

- Both surfaces expose **previous** and **next** file-in-folder navigation with consistent ordering and hidden-file rules.
- No navigation into directories via these controls; only **files**.
- Dirty-state and save/discard behavior matches current single-click open behavior.
- Controls are keyboard-focusable and labeled for accessibility.
- Brief note in `fixes/` or roadmap cross-link when done (optional one-liner in MASTER plan).
