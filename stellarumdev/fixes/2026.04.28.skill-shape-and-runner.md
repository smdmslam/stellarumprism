# prism-skill-v1 \u2014 a shape contract for executable skills
A follow-up captured from the discussion around the recipe runner +
ProtocolReportCard. Once we had the card and the runner, the natural
next question was: can the same lifecycle drive an LLM working through
a defined plan, not just a hand-coded recipe? The answer is yes \u2014 if
the plan is authored against a contract.

## Status
Not started. Phase B + Phase C of the protocol-recipes work shipped
(`500134e`, `f688294`); this doc extends the same execution model to
user-authored skills.

## Background
Skills today live in `.prism/skills/*.md` and are loose prose. They're
useful as agent context (when wiring lands per
`reviewer-methodology.md` Phase 3) but not directly executable \u2014 the
agent has to reinterpret them every time, and there's no way to watch
the work progress as ordered steps.

The Claude / Anthropic skills ecosystem has a growing community
authoring `.md` skills, and the loose-Markdown shape is genuinely
useful for human transmission. We do not need to replace it. What we
DO need is an opt-in structured layer for skills that should drive the
ProtocolReportCard the same way recipes do today: deterministic step
list, live progress bullets, on-disk report, cancellable.

The win: every system gets to do what it does best.
* **Loose-prose skills** stay loose-prose \u2014 they read like a runbook,
  ship in a community library, get loaded into the agent as context
  for fuzzy / open-ended work. No card, no opinion, no friction.
* **Structured skills** (those that opt in to `prism-skill-v1`) get
  the same lifecycle UX recipes already have: planning state with the
  full step list, running state with live bullet transitions, done
  state with a saved report. Same component, same plumbing.

Systemization for machine consumption isn't a value judgement on
loose-prose. It's a different consumption mode \u2014 runbook-as-data
vs runbook-as-prose \u2014 and there's room for both.

## The contract: prism-skill-v1

A skill conforms to the shape if and only if:

1. **Frontmatter declares it.** YAML frontmatter at the top includes
   `shape: prism-skill-v1`.
2. **A `## Steps` section exists.** Anywhere in the file. Contains a
   numbered list. Each item follows: `**short title** \u2014 prompt body
   with optional `{{var}}` substitutions`.
3. **Inputs are declared.** Frontmatter `inputs:` lists every variable
   referenced in the steps. Each input has `name` and a short
   description so the runner can prompt the user (or the calling
   skill) to supply values at invocation.
4. **Optional metadata.** `title`, `description`, `category` (matches
   the recipe `category` enum), `default_model`, `tags`.

Anything else in the file (`## Background`, `## Examples`, raw prose,
embedded code blocks) is free-form context. The skill loader passes
that body to the agent as a system-prefix during execution so the
model has the surrounding rationale, not just the step list.

### Example skill file

```markdown path=null start=null
---
title: VC Pitch Deck Template
description: Generate a firm-specific pitch deck template based on
  the firm's public preferences and portfolio patterns.
shape: prism-skill-v1
category: research
inputs:
  - name: firm
    description: Name of the VC firm (e.g. "Sequoia Capital")
default_model: x-ai/grok-4.1-fast
---

## Steps

1. **Research investment thesis** \u2014 Search the web for `{{firm}}
   investment thesis 2024-2026` and synthesize 3-5 bullet points
   covering stage focus, sector themes, and check-size patterns.
2. **Survey recent portfolio** \u2014 Search for `{{firm}} recent
   investments 2025` and pull 8-10 representative companies with
   sectors and stages.
3. **Identify deck patterns** \u2014 From the gathered material,
   synthesize: which slides this firm responds to best, what
   narrative beats they expect, what data density they prefer.
4. **Generate template** \u2014 Write a Markdown pitch-deck template
   with sections matching the firm's apparent preferences. Include
   placeholder copy and structural guidance per slide.
5. **Save template** \u2014 Write the template to
   `pitch-decks/{{firm | slugify}}-template.md`.

## Background

Bootstrapped by working through a16z's apparent preferences in a
chat session, then formalized so the same flow applies to any firm.
Lessons learned from that bootstrap:
- Search results often conflate investment thesis with portfolio
  highlight reels; step 1 explicitly asks for the thesis.
- Some firms have public deck templates (Sequoia famously does);
  step 3 should mention this when found.
```

## Skill loader: shape \u2192 Recipe

A new module `src/skills/loader.ts`:
* Reads a skill .md file via `read_file_text`.
* Parses YAML frontmatter (small dependency or a tiny in-house parser).
* Validates against `prism-skill-v1`.
* If valid, returns a `Recipe` shape (the same type recipes use today)
  with one `prompt` step per numbered list item, body interpolated with
  the user-supplied input values.
* If not valid, returns a `loose` flag plus the raw skill content; the
  caller decides whether to inject as context or surface a "not
  enumerated" notice.

```ts path=null start=null
type LoadedSkill =
  | { kind: "structured"; recipe: Recipe; rawContext: string }
  | { kind: "loose"; rawContent: string; missing: string[] };

async function loadSkill(
  pathOrId: string,
  inputs: Record<string, string>,
): Promise<LoadedSkill>;
```

`rawContext` is the non-Steps prose; the runner injects it as a
per-turn `systemPrefix` (the same channel Grounded-Chat uses today)
so the model sees the rationale without it bloating session history.

## New runner step kind: `prompt`

`StepDef` gains a third variant:

```ts path=null start=null
| {
    kind: "prompt";
    label: string;
    /** Prompt body, with {{var}} already substituted. */
    prompt: string;
    /** Optional model override for this step (mirrors Mode pattern). */
    modelOverride?: string;
    onFailure?: "abort" | "continue";
  };
```

Runner adds a sibling to `runShellStep` / `runSlashStep`:

```ts path=null start=null
async function runPromptStep(
  def: Extract<StepDef, { kind: "prompt" }>,
  ctx: RecipeRunnerCtx,
): Promise<StepResult>;
```

Implementation reuses `ctx.runSlashCommand` semantics: dispatch a
turn (no slash command needed; just a raw prompt), await
`agent.awaitTurnComplete`, capture the assistant response into
`StepResult.output`. The `runSlashCommand` helper might split into
`runSlashCommand` + `runRawPrompt` (cleaner) or grow a second arg
(simpler). Pick during implementation.

## Skill validator (`/skill check <id|path>`)

A new slash command for skill authors:

* Loads the file.
* Reports: shape declared? Steps section parseable? Every input
  declared and used? Every `{{var}}` referenced in steps declared as
  an input? Any leftover free-form sections that the runner will pass
  as context?
* Output via `appendReport(markdown)` so the validator's findings
  render as a styled section, not a flat notice.
* Exit cleanly when valid; print a checklist of issues otherwise.

This is the `prism-audit` of skills \u2014 same pattern as the
substrate-cell-driven validation we use elsewhere.

## Skill builder (loose .md \u2192 prism-skill-v1)

Two flavors, both useful:

### Flavor A: manual converter
A new slash command `/skill convert <path>`:

* Loads a loose .md skill.
* Heuristically extracts step candidates: numbered lists, headings
  shaped like "Step N", verb-led bullet lists.
* Asks the user (via the modal text-input pattern already in
  `askText`) to confirm + edit the step list, label each step, name
  the inputs.
* Writes the result to `<path>.prism-skill-v1.md` (or back to the
  original with a backup) with the frontmatter + Steps section
  prepended; the original prose moves into `## Background`.
* Validates the result with the validator above before writing.

### Flavor B: agent-assisted
A new mode `skill-build`:

* User invokes `/skill build <description>` or
  `/skill build --from <path>`.
* Mode persona's system prompt: "You are a skill author. Take the
  user's description (or the loose .md provided) and produce a
  conforming `prism-skill-v1` file. Output ONLY the markdown."
* Result lands as a proposed skill file the user can save with one
  click (or approve the proposed write through the existing approval
  flow).

Flavor A is deterministic and safe; Flavor B is faster but model-
dependent. Ship A first; B is opt-in polish.

## Backwards compatibility

* Non-conforming skills KEEP working as agent context. When the user
  invokes a skill (via the future skills toggle in Settings or a
  future `/skill run <id>` command), the loader runs:
  * Structured \u2192 mount the card, run the recipe.
  * Loose \u2192 inject the file content as a per-turn `systemPrefix`,
    print a small notice: `[skill] running "<id>" as freeform context
    \u2014 not enumerated, no card. /skill convert <id> to upgrade.`
* No existing skill in `.prism/skills/` becomes invalid; users only
  see card UX when they opt in by adding the frontmatter + `## Steps`.

## Acceptance checks

* A `prism-skill-v1` skill in `.prism/skills/` invoked via `/skill run
  <id>` mounts a ProtocolReportCard, runs each step as an agent turn,
  marks bullets ok / failed live, writes a markdown report, opens it
  on click.
* Loose-prose skills run as before \u2014 no card, no break, just
  context.
* `/skill check <path>` reports specific shape violations as a
  styled report block.
* `/skill convert <path>` produces a valid `prism-skill-v1` file
  (validator passes on the output).
* Skill cancellation propagates: cancel button on the card aborts at
  the next step boundary, just like recipes.
* Skill `inputs` get supplied via a small modal (or via flag args
  passed to the slash command, e.g. `/skill run vc-deck firm=Sequoia`).

## Phasing / commit shape

Each piece is small enough to ship independently:

1. **Skill types + frontmatter parser.** `src/skills/types.ts`,
   `src/skills/parser.ts`. Pure module; no UI.
2. **Skill loader + validator.** `src/skills/loader.ts`. Returns
   `LoadedSkill`. New slash command `/skill check` wired.
3. **`prompt` step kind in the runner.** `src/recipes/types.ts`
   gains the variant; `runner.ts` gains `runPromptStep`.
4. **`/skill run <id>` slash command.** Wires loader \u2192 runner \u2192
   ProtocolReportCard with the existing infrastructure.
5. **Skill builder (Flavor A: manual converter).** `/skill convert`
   slash command + heuristic extractor + validator round-trip.
6. **Skill builder (Flavor B: agent-assisted, opt-in).** New mode
   `skill-build`; persona prompt; approval-flow integration.

## Risks

* **Frontmatter parsing.** YAML in the wild has gotchas (multi-line
  strings, nested objects). Keep the parser narrow: only the keys we
  defined, fail loud on anything else. A 200-line hand-roll covers
  what we need without pulling in `js-yaml`.
* **`{{var}}` substitution.** Simple string replace works for v1.
  Watch for accidental collisions with Markdown that legitimately
  contains `{{...}}` (rare but possible). Escape mechanism needed
  if it surfaces.
* **Model reliability per step.** Some steps may not produce a clear
  "done" signal in their assistant output; the runner should treat any
  non-error agent turn as ok and let the user judge from the report.
  If recipes need stricter completion semantics, add a per-step
  `success_pattern` regex to the shape (v2).
* **Skill drift on shape upgrades.** If we ever move to
  `prism-skill-v2`, the validator should still recognize v1 skills
  and run them with v1 semantics. Version the shape, not the runner.

## Out of scope

* Cross-skill composition (one skill calling another). Probably
  natural to add later; same way recipes might want to run other
  recipes. Not v1.
* Mid-run plan mutation (model decides step 3 isn't needed). The
  whole value of the shape is determinism; if the model wants to
  improvise, that's the freeform-context path.
* Skill marketplace / shared-skill installation. Filesystem-only for
  now; users author in `.prism/skills/`.
* Replacing loose-prose skills. They're a peer, not a predecessor.

## Why this is its own session

The recipe runner gives us the lifecycle infrastructure for free.
The work here is wiring + a small parser + one new step kind. Total
scope reads like 6-8 hours of focused work, sequenced as tight per-
piece commits. Larger if Flavor B (agent-assisted builder) is
included up front; small if we ship Flavor A first and treat B as
later polish.

## Why this matters
Loose-prose skills are great for human transmission. Structured
skills are great for machine execution. Skills that need to be both
get to be both via the contract \u2014 same file, conforming sections
plus free-form sections \u2014 with no value judgement on which mode is
"better". The shape is a tool, not a mandate.

The bigger lesson the design discussion surfaced: **the card and
runner are general enough to drive recipes today and skills next, and
the only piece that needs to evolve is the data model that feeds
them**. That's the right kind of architectural lever to lean on.
