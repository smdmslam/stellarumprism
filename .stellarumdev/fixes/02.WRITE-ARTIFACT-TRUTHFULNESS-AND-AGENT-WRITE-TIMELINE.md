# 02.WRITE-ARTIFACT-TRUTHFULNESS-AND-AGENT-WRITE-TIMELINE.md

**Status:** Draft implementation plan  
**Intent:** Fix the misleading write/diff experience in Prism by separating **approval previews** from **post-execution artifacts**, enriching the `agent-tool-*` event contract, and rendering a truthful per-turn write timeline for important edits.

**Related evidence:** `src/agent.ts`, `src/agent-view.ts`, `src/turn-summary.ts`, `src-tauri/src/agent.rs`, `src-tauri/src/tools.rs`, `docs/agent-diff-ui-test.ts`

---

## 1. Problem statement

Prism currently shows inline write diffs by reusing the **approval preview** captured before a tool executes, rather than a **canonical post-execution artifact**.

This causes three user-trust failures:

1. **`write_file` overwrite looks like pure addition.**  
   A file overwrite is rendered using the `write_file:` preview body, which contains only new content, so the diff card becomes an all-green added-only view even when the operation conceptually replaced existing content.

2. **Failed edit attempts are not coherently explained.**  
   The user may see an `edit_file` failure (`old_string not found`) and then a successful `write_file`, but Prism does not present those as one causal sequence. The user is left to infer what happened.

3. **The event contract drops backend truth.**  
   Rust already has a structured `ToolInvocation { ok, summary, payload }`, and write-tool payloads already know facts like `created`, but `agent-tool-*` currently emits only `name`, `args`, `summary`, `ok`, `round`.

The result is a product that can appear more certain and more precise than it really was during an important edit.

---

## 2. Verified current behavior

### 2.1 Approval preview is cached and later reused as the diff card

In `src/agent.ts`, `onToolApproval()` stores the preview in `lastApprovalDiff`, and `onToolCall()` later renders that cached string with `appendDiff(...)` when the write succeeds.

Implication: the final diff card is not a post-write artifact; it is a recycled approval artifact.

### 2.2 `agent-tool-*` completion event is too thin

In `src-tauri/src/agent.rs`, `agent-tool-*` emits:
- `name`
- `args`
- `summary`
- `ok`
- `round`

It does **not** emit:
- `call_id`
- `payload`
- operation kind
- actual diff artifact

Implication: the frontend cannot reliably correlate approval → completion, nor can it inspect write-tool payloads that already exist in Rust.

### 2.3 `write_file` and `edit_file` previews have different truth levels

In `src-tauri/src/tools.rs`:
- `preview_write("edit_file", ...)` returns `--- old` / `+++ new`
- `preview_write("write_file", ...)` returns only the new content body

In `src/agent-view.ts`:
- `parsePrismEditPreview()` creates removed + added lines
- `parsePrismWritePreview()` creates only added lines

Implication: added-only rendering is correct for *new-file creation*, but misleading for *overwrite-existing-file*.

### 2.4 Footer stats also flatten overwrite semantics

In `src/turn-summary.ts`, `calculateWriteStats()` treats:
- `edit_file` as old vs new line counts
- `write_file` as `added = all new lines`, `removed = 0`

Implication: overwrite-via-`write_file` is misrepresented in both the diff card and the files-modified footer.

### 2.5 Exact-copy discipline already exists in repo policy

The repository already documents the correct behavioral rule:
- always `read_file` first
- use `edit_file` for small targeted edits
- copy exact `old_string` from read output
- retyping causes `old_string not found`

Implication: this fix is not inventing a new philosophy; it is aligning the product with its existing safety model.

---

## 3. Design goals

The fix should produce five guarantees:

1. **Approval preview is clearly marked as preview.**
2. **Post-execution UI is based on post-execution truth whenever available.**
3. **The frontend can correlate one approval with one completion deterministically.**
4. **Create vs overwrite vs edit are visibly distinguished.**
5. **The user can see the write attempt sequence, including failures.**

---

## 4. Scope for this implementation

This plan is intentionally split into phases so we can land user-trust improvements early without waiting for full overwrite-diff generation.

### Phase 1 — contract + truthful labeling
- Enrich `agent-tool-*` with `call_id` and `payload`
- Replace `lastApprovalDiff` with a `call_id`-keyed map
- Distinguish approval preview from actual artifact in the frontend
- Label `write_file` completion as **create** vs **overwrite** using payload
- Render a first-pass write timeline

### Phase 2 — actual overwrite artifacts
- Extend backend write tools to emit canonical old/new or unified diff when overwriting existing text files
- Upgrade diff cards to use actual overwrite artifacts instead of preview fallback

### Phase 3 — stricter behavioral guardrails
- Require reread after `old_string not found`
- Harden overwrite approvals for existing files
- Optionally block silent fallback from failed `edit_file` to `write_file` on same path without explicit overwrite semantics

---

## 5. Implementation plan

## 5.1 Rust: enrich the `agent-tool-*` completion contract

**File:** `src-tauri/src/agent.rs`

### Current emitter
The completion event currently emits only:
- `name`
- `args`
- `summary`
- `ok`
- `round`

### Change
Emit:
- `call_id`
- `name`
- `args`
- `summary`
- `ok`
- `round`
- `payload`

### Why
- `call_id` gives deterministic correlation with `agent-tool-approval-*`
- `payload` exposes Rust-side truth already computed by `ToolInvocation`

### Acceptance
- Existing frontend still works after a TS-side listener update
- No behavior change for non-write tools except richer metadata availability

---

## 5.2 TypeScript: replace `lastApprovalDiff` with `pendingApprovals`

**File:** `src/agent.ts`

### Current state
A single `lastApprovalDiff: string | null` is used globally for the turn.

### Change
Replace with something like:

```ts
private pendingApprovals = new Map<string, {
  tool: string;
  args: string;
  preview: string;
  round: number;
}>();
```

Store entries in `onToolApproval(info)` keyed by `info.call_id`.

When `onToolCall(info)` fires:
- look up `pendingApprovals.get(info.call_id)`
- use that approval metadata only for the matching completion
- delete the map entry after completion is handled

Lifecycle / map-management requirements:
- clear `pendingApprovals` at the start of every new user turn
- clear any surviving entries on terminal turn states like `onDone()` and `onError()`
- do not leave unmatched approval entries resident after completion or cancellation

### Why
This removes the implicit “latest approval wins” coupling and makes approval/completion pairing explicit.

### Acceptance
- Two consecutive approvals cannot cross-wire their diff cards
- `read_skill` and other approval-gated tools still work normally
- stale approval entries do not accumulate across turns or failed requests

---

## 5.3 TypeScript: widen listener types and callback surfaces

**Files:** `src/agent.ts`, possibly `src/workspace.ts`

### Change
Update the `agent-tool-*` listener type to include:
- `call_id: string`
- `payload: string`

Update `onToolCall(info)` signature accordingly.

When parsing `payload` in `src/agent.ts`, require defensive parsing:
- wrap `JSON.parse(info.payload)` in `try/catch`
- treat malformed payload as a degraded-but-non-fatal path
- fall back to summary-only rendering when payload cannot be trusted

Optionally widen `onToolExecuted` to pass through:
- `call_id`
- `payload`

while preserving the currently used fields:
- `name`
- `args`
- `ok`

### Why
This keeps downstream hooks future-proof while preserving current `read_skill` behavior.

### Acceptance
- `workspace.ts` continues to engage `read_skill` correctly
- No downstream compile errors from contract widening

---

## 5.4 Frontend: stop pretending preview == final artifact

**Files:** `src/agent.ts`, `src/agent-view.ts`

### Current state
`appendDiff(path, diff)` receives raw text only.

### Change
Introduce a richer API shape, e.g.:

```ts
appendDiff(info: {
  path: string;
  diff: string;
  source: "approval-preview" | "tool-artifact";
  operation?: "create" | "overwrite" | "edit";
})
```

During Phase 1:
- successful `edit_file` can still use the matched approval preview, but label it `preview`
- successful `write_file` with payload `created: true` can render as `Created file · preview`
- successful `write_file` with payload `created: false` must render as `Overwrote existing file · preview only`
- overwrite-preview cards should carry a subtle warning-style visual treatment (e.g. warning glyph, dim caution text, or italicized `preview only`) so the user immediately understands they are not seeing a true line-by-line diff yet

### Why
Even before real overwrite diffs exist, users must be told what kind of artifact they are looking at.

### Acceptance
- Diff card header distinguishes create vs overwrite vs edit
- Overwrite case does not masquerade as a true diff
- `preview only` state is visually clearer than a normal success diff card

---

## 5.5 Frontend: add a write-attempt timeline block

**Files:** `src/agent.ts`, `src/agent-view.ts`, maybe `src/turn-summary.ts`

### Change
Track each write attempt as a structured record:
- `call_id`
- `tool`
- `path`
- `ok`
- `summary`
- `operation` (if known)
- `preview_source` / `artifact_source`

Render a simple ordered list near the end of the turn, e.g.:
- `edit_file docs/test-poem.md — failed: old_string not found`
- `write_file docs/test-poem.md — created file`

Initial version can be plain text / chip style.

### Why
This gives users the missing execution narrative and directly addresses the trust gap.

### Acceptance
- Failed write attempts are visible in sequence, not only as isolated tool cards
- A later successful fallback is visibly connected as a second attempt

---

## 5.6 Frontend: improve footer truthfulness for `write_file`

**File:** `src/turn-summary.ts`

### Current state
`write_file` always reports `removed: 0`.

### Phase 1 change
Do not fabricate removals if we still lack old text, but avoid overstating certainty.

Options:
1. keep `+N` only for true creates, and suppress line stats for overwrites until actual diff exists
2. or add an overwrite sentinel shape so chips can show `overwrite` instead of fake `+N -0`

### Recommended
For `write_file` where payload says `created: false`, suppress added/removed stats in Phase 1 and label the chip as overwrite-only.

### Why
Unknown removal counts are better than false removal counts.

### Acceptance
- Footer/chips no longer imply overwrite == additions-only

---

## 5.7 Backend Phase 2: emit actual overwrite artifacts

**File:** `src-tauri/src/tools.rs`

### Current state
`tool_write_file()` knows:
- whether file existed
- old size
- new size

but does not preserve old content.

### Change
For text-file overwrites:
- read previous UTF-8 content before writing
- compute either:
  - `old_text` + `new_text`, or
  - a canonical unified diff
- include that in payload for `write_file`

Potential payload extension:

```json
{
  "path": "...",
  "created": false,
  "bytes_written": 123,
  "previous_bytes": 98,
  "artifact": {
    "kind": "write",
    "operation": "overwrite",
    "diff": "..."
  }
}
```

### Why
This is the first moment Prism can render a truthful overwrite diff card.

### Acceptance
- Overwriting an existing text file yields a real diff artifact
- New file creation still yields create semantics
- Binary/non-UTF8 edge cases degrade gracefully (no fake diff)

---

## 5.8 Backend/behavior Phase 3: force reread after failed `edit_file`

**Files:** likely `src-tauri/src/agent.rs` prompt/rules and/or agent-side control logic

### Change
If `edit_file` fails with `old_string not found` for a path:
- require a fresh `read_file` on that path before another write attempt to the same file in the same turn

Optional stronger rule:
- if the same path already existed, do not allow fallback to `write_file` without an explicitly dangerous overwrite path in the approval UX

### Why
This directly targets the brittle behavior the repository already warns about.

### Acceptance
- agent cannot thrash on a stale/guessed edit target without rereading exact content
- same-turn fallback becomes explicit and safer

## 6.2 Manual verification scenarios

### Scenario A — exact `edit_file`
1. ask agent to change a uniquely matched line
2. approve `edit_file`
3. verify card says `Edited file`
4. verify diff shows old/new preview semantics

### Scenario B — create new file
1. ask agent to create a brand-new file
2. approve `write_file`
3. verify card says `Created file`
4. added-only rendering is acceptable

### Scenario C — overwrite existing file
1. ask agent to replace an existing file via `write_file`
2. approve
3. Phase 1 expectation: card says `Overwrote existing file · preview only`
4. Phase 2 expectation: card shows actual overwrite diff

### Scenario D — failed edit then fallback
1. deliberately trigger `old_string not found`
2. observe failed timeline row
3. if a later write succeeds, observe a second timeline row with create/overwrite semantics
4. verify UI no longer flattens this into one apparently clean diff

## 7. Risks / edge cases

### 7.1 Binary or non-UTF8 overwrite
Do not attempt textual diff generation. Render:
- `Overwrote existing file`
- `binary or non-text artifact; no inline diff`

### 7.2 Large overwrite payloads
Actual old/new content can be large. Phase 2 should use a bounded diff artifact, not unlimited payload growth.

### 7.3 Backward compatibility
Keep `summary` and current display paths intact while new fields land, so existing tool cards remain stable during migration.

### 7.4 Saved transcript parity
This plan fixes live event truthfulness first. Historical transcript replay may later want richer persisted write artifacts, but that is not required for the first user-trust win.

---

## 8. Recommended execution order

1. **Event contract patch** (`call_id`, `payload`)  
2. **Frontend approval/completion correlation refactor**  
3. **Preview-vs-actual labeling in diff cards**  
4. **Write timeline block**  
5. **Footer/chip truthfulness adjustment for overwrites**  
6. **Backend actual overwrite artifacts**  
7. **Reread-after-failed-edit guardrail**

This order gives immediate user-facing trust improvements before the more invasive overwrite-diff work.

---

## 9. Acceptance criteria

This fix is complete when all of the following are true:

- [ ] `agent-tool-*` emits `call_id` and `payload`
- [ ] frontend correlates approvals and completions by `call_id`, not by a single global cached string
- [ ] diff cards explicitly label preview vs actual artifact provenance
- [ ] `write_file` completion visibly distinguishes **create** vs **overwrite**
- [ ] failed write attempts appear in a coherent per-turn timeline
- [ ] overwrite-via-`write_file` no longer presents as a clean all-green diff without warning
- [ ] Phase 2: existing-file `write_file` produces a real overwrite artifact when text content is available
- [ ] Phase 3: failed `edit_file` requires reread before retrying same-path write logic

---

## 10. Proposed first implementation slice

If we want the smallest useful first session, do exactly this:

1. patch Rust `agent-tool-*` to emit `call_id` + `payload`
2. patch TS listener and `onToolCall()` to accept them
3. replace `lastApprovalDiff` with `pendingApprovals: Map<call_id, ...>`
4. change `appendDiff()` signature to include `source` + `operation`
5. label `write_file` completion as:
   - `Created file · preview`
   - `Overwrote existing file · preview only`

That slice does **not** solve real overwrite diffing yet, but it immediately removes the most misleading behavior and establishes the correct architecture for the rest.
