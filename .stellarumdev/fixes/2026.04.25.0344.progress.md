Where Prism is right now
You have a working diagnostic substrate + three consumers running inside a Tauri shell with an xterm + CodeMirror surface. That’s real. Specifically:

Substrate cells (deterministic, LLM-free):
•  typecheck — compiler ground truth (tsc / cargo / go / pyright)
•  ast_query — symbol resolution via the TS compiler API
•  run_tests — project test suite, parsed failures
•  http_fetch — live endpoint probe (just shipped)

Consumers (LLM-driven, substrate-gated):
•  /audit — compiler-first verifier, confidence-graded findings, JSON sidecar
•  /fix — applies findings from a sidecar through the approval flow
•  /build — plan → baseline → execute (read+edit+verify) → iterate → BUILD REPORT, with substrate gates between every edit

Glue that exists:
•  Approval flow on every write
•  Confidence grader (confirmed / probable / candidate) keyed off evidence source
•  Tool-round budgets per mode, model selection per mode
•  Reviewer pass after primary completes
•  Image attachments, slash commands, mode registry, JSON sidecars

Distance to “bonafide IDE that helps users build apps”
Honest answer: you’re maybe 35–45% there. The hard, differentiated half is mostly built. The remaining half is partly differentiated, partly table stakes that users will assume exist.

I’ll break it into three buckets.

1. The substrate (mostly done, ~75%)
What’s still missing:
•  LSP cell. You have ast_query for TS, but a generic LSP bridge would unlock hover, go-to-def, find-references, rename, and per-language diagnostics for free across Rust/Python/Go. This is high-leverage; one cell, ten tools.
•  A thin E2E runner. http_fetch is one round-trip; a recorded-flow probe (login → action → assert) closes the gap to "the feature actually works." This is substrate v5 in the prior plan.
•  Optional: telemetry/log probe. Tail the dev server logs and parse errors. Catches things http_fetch 200-OKs but the server logged a crash for.

2. The consumers / IDE surface (early, ~25%)
This is where you’re furthest from "IDE." Today the user lives in xterm-with-slash-commands. Real IDE-ness needs:

•  A files panel + editor that understands the substrate. Diagnostics inline as squiggles, sourced from the same Diagnostic shape typecheck already emits. The substrate is shaped right; the renderer doesn’t exist.
•  A problems panel showing audit findings live, grouped by file, click-to-jump.
•  A diff/approval review surface that isn’t a one-line preview. Real syntax-highlighted side-by-side, with “approve hunk” granularity. Today it’s a single yes/no per write.
•  A run/debug surface. Even just “run dev server” with output capture and a stop button turns it from a chat with tools into an environment.
•  Project-aware command palette + go-to-anywhere. ast_query already has the data; needs a UI.
•  Persistent workspace state. Recent files, breadcrumbs, last audit, last build report — survives reload.

3. App-building workflows (almost nothing yet, ~10%)
This is the long tail that turns "AI-assisted editor" into "AI helps users build apps." None of these exist yet:

•  Project bootstrap (/new). "Build me a SaaS with auth and Stripe" → scaffolded via templates + substrate verification per step. /build is the engine; this is the on-ramp.
•  Schema + migration awareness. Detect a Prisma/Drizzle/SQLAlchemy schema, route migrations through the approval flow, run them as a substrate cell.
•  Dependency management as a substrate operation. npm install / cargo add with version pinning and changelog summary, gated.
•  Deploy/preview integration. A "ship it" button that runs typecheck + tests + build + deploy-preview. Vercel/Fly/Render adapters.
•  Real multi-file refactors. /refactor rename Foo to Bar across the project driven by ast_query + grader. The substrate supports this; no consumer wires it yet.

What I’d build next, in order
Smallest leverage-multipliers first:

1. Auto-detect dev server URL (the next step from yesterday). Tiny config knob, makes http_fetch actually usable without prompting hacks.
2. Surface http_fetch results in the audit JSON sidecar so /fix picks up runtime-graded findings cleanly.
3. LSP cell (substrate). Single biggest leverage move — turns Prism from TS-first into language-agnostic in one PR.
4. Problems panel UI (consumer). The substrate already emits the right shape; this is mostly a renderer. First piece of "this feels like an IDE."
5. Inline editor diagnostics (consumer). Same data, different render target. Once it’s in the panel, putting it in CodeMirror as squiggles is small.
6. /refactor (consumer). Reuses ast_query + grader + approval flow. Demonstrates that the substrate generalizes beyond audit/fix/build.
7. /new <feature> scaffolder (consumer). The on-ramp.
8. E2E runner (substrate v5). Closes the runtime-evidence gap for stateful flows.

Steps 1–2 are hours. Step 3 is days. Step 4 is the first one that visibly ships “IDE”-ness to a user. Steps 7–8 are the ones that get you to “bonafide app builder” instead of “very good audit/fix tool.”

Honest caveat
The thing you have built — substrate-gated AI editing — is the right architecture for this product, and most competitors haven't figured this out yet. But "right architecture" is invisible to users until you put it behind an IDE-shaped surface. The bottleneck from here is frontend velocity, not substrate sophistication. You have more substrate than you have UI to expose it.

Pick item 4 (problems panel) when you’re ready to start making the substrate user-visible. That’s the inflection point where outsiders will look at Prism and say "oh, that’s an IDE."
