# Agent transport & production architecture

**Status:** Draft (ADR-style)  
**Intent:** Define how Prism’s **LLM traffic** and **user trust model** should evolve from today’s **local OpenRouter (BYOK)** configuration toward **industry-standard** patterns (managed account + server-held keys, optional BYOK), without throwing away the existing Tauri + Rust `agent` stack.

**Related:** [BETA-DISTRIBUTION-PLAN.md](./BETA-DISTRIBUTION-PLAN.md), `src-tauri/src/agent.rs`, `src-tauri/src/config.rs`, `src-tauri/src/billing.rs`, `src/services/firebase.ts`

---

## 1. Context — what exists today

| Layer | Current behavior (summary) |
|--------|----------------------------|
| **Runtime** | Tauri 2 desktop; heavy lifting in **Rust** (`agent_query`, streaming SSE, tools). |
| **LLM provider** | **OpenRouter** — API key from user config: `~/.config/prism/config.toml` → `[openrouter] api_key`. |
| **Wire format** | OpenAI-style tools + streaming; same shape many providers support. |
| **Billing / tier** | Rust `billing` module, subscription state on disk, usage deduction hooks, UI surfaces (`get_subscription_info`, etc.). |
| **Frontend** | Firebase client present for some flows (e.g. usage persistence); not yet the single gate for *all* agent traffic. |

**Implication:** Prism is already a **real product shell**; the main architectural fork for “mass market” is **where the API key lives** and **how the client authenticates**—not whether to use HTTP or tools.

---

## 2. Industry pattern (reference, not novel)

Mainstream IDE / dev tools (same category as Cursor-class products) typically use:

1. **User identity** (account) — not a raw provider key in `~/.config` for the default path.
2. **Entitlements** — what plan, which models, quotas, kill switch.
3. **Server-side provider keys** — your infra calls OpenRouter / Anthropic / etc.; the desktop holds **session tokens**, not **provider secrets**.
4. **Optional BYOK** — advanced users paste their own key; traffic may bypass your proxy or use a hybrid policy.

This document calls that **managed transport**. Prism’s current path is **direct BYOK transport**.

## 3. Two transports (both first-class in design)

### 3.1 Transport A — **Direct BYOK** (today)

- **Flow:** Rust reads `api_key` (+ optional `base_url`) from config → connects to OpenRouter (or compatible endpoint).
- **Pros:** Simple, no backend dependency, good for power users and internal dogfood.
- **Cons:** Key on disk; every tester must supply a key; org cannot centrally pay or revoke without rotating keys yourself.

**Keep as:** Supported mode indefinitely for developers and opt-in users.

### 3.2 Transport B — **Managed API** (target for mass distribution)

- **Flow:** User signs in (e.g. Firebase Auth or equivalent). Desktop obtains **short-lived access tokens**. Agent HTTP layer sends requests to **`PRISM_API_BASE`** (HTTPS), not directly to OpenRouter with a user key.
- **Server:** Validates JWT / session, checks entitlements, attaches **your** provider credentials server-side, proxies streaming responses (or uses a gateway). Rate limits, audit, abuse controls live here.
- **Pros:** Same trust model as web SaaS; keys never ship to clients; billing and revocation are coherent.
- **Cons:** You operate a secure API; latency and reliability become shared concerns.

**Naming:** Environment/config examples — `PRISM_API_BASE`, `PRISM_AUTH_*` — are illustrative; final names should match one convention (`tauri.conf` + Rust config merge).

## 4. Prism-specific design sketch

### 4.1 Single agent core, pluggable HTTP backend

The **tool loop, approval flow, streaming events, and substrate** stay in **Rust**. Only the **HTTP client configuration** branches:

| Concern | BYOK | Managed |
|---------|------|---------|
| Base URL | OpenRouter default / config `base_url` | Your API origin |
| Auth header | `Authorization: Bearer sk-or-...` | `Authorization: Bearer <user_access_token>` |
| Model slug | Passed through (OpenRouter slug or mapped server-side) | Same wire shape; server may map aliases |
| Usage accounting | Existing local billing hooks | Prefer **server-authoritative** usage + sync or hybrid |

**Implementation direction:** Introduce a small internal trait or enum in `agent.rs` (or a submodule) — e.g. `LlmTransport` — implemented by `OpenRouterDirect` vs `PrismManagedProxy`. **Do not** fork two agent loops.

### 4.2 Configuration precedence (recommended)

1. If **managed mode enabled** (user signed in + feature flag / setting): use Transport B.
2. Else if **`[openrouter] api_key` present**: use Transport A.
3. Else: agent disabled with a **single** clear UX path (wizard → sign in **or** paste key).

Avoid silent fallback that mixes modes without user clarity.

### 4.3 Billing alignment

Today: Rust deducts usage based on token/cost signals from the stream.

For managed transport:

- **Authoritative spend** should ultimately come from **your backend** (or agreement between backend meter and client display).
- Client-side deduction may remain as **optimistic UI** until confirmed, or be replaced by server-reported balances — decide explicitly to avoid double-charge / drift bugs.

Document the chosen rule in the same PR that wires managed transport.

### 4.4 Secrets

- **Never** ship provider keys in the desktop bundle or user `.env` for production testers.
- `.env` remains **developer / CI only** for Firebase config, dev API URLs, etc.

## 5. Phasing (recommended order)

| Phase | Deliverable | Unlocks |
|-------|-------------|---------|
| **P0 — Ship** | Signed installers, updater, version + git SHA in UI, first-run messaging, layout recovery doc | Trustworthy binaries and supportable bug reports |
| **P1 — Identity** | Sign-in UI + token storage (secure store), “signed out” vs “signed in” states | Foundation for managed mode without changing LLM wire yet |
| **P2 — Managed MVP** | Minimal HTTPS proxy: auth + forward streaming completion + map errors; feature flag | Mass beta without BYOK |
| **P3 — Harden** | Rate limits, observability, billing reconciliation, optional BYOK UX polish | Production posture |

Parallel track: **BYOK** remains testable on every phase via config.toml.

---

## 6. Security checklist (non-exhaustive)

- TLS only; certificate pinning optional later.
- Short-lived access tokens + refresh; revoke on logout.
- No provider keys in logs, crash dumps, or client payloads.
- Server-side validation of model allowlists per tier.
- Privacy policy + ToS once accounts collect PII.

---

## 7. Open decisions (to resolve before implementation tickets)

1. **Protocol:** Same JSON/SSE shape end-to-end vs thin wrapper around existing OpenRouter-compatible API?
2. **Auth:** Firebase Auth only vs add others later?
3. **BYOK when signed in:** Forbidden, allowed with warning, or separate “advanced” panel?
4. **Offline:** Explicit degraded mode (shell/editor only) when managed API unreachable?

---

## 8. Acceptance criteria for “this ADR is implemented”

- [ ] One Rust **transport abstraction**; two implementations (direct + managed).
- [ ] User-visible **mode** is explainable in one sentence on first launch.
- [ ] No production reliance on users editing `.env` for LLM access.
- [ ] **Managed** path never places OpenRouter **secret** keys on client disk.
- [ ] Docs updated: README + beta tester path reflects download → install → sign in **or** BYOK.
